# 资源加载之后会经历什么

很多人把下面这句代码理解成“资源已经能直接用了”：

```ts
const res = await Laya.loader.load("resources/ui/logo.png");
```

但 `load()` 真正完成的，是“把指定 URL 对应的资源经过识别、下载、解析后，交还给运行时”。

这中间至少还分成五步：

1. 识别 URL 和资源类型
2. 选择对应 loader
3. 处理缓存和并发去重
4. 解析成资源对象或运行时可直接使用的对象
5. 如果是层级资源，再继续反序列化成运行时对象树

如果这五步没分开看，就很容易把下面几件事混成一件事：

1. 文件是否下载到了
2. loader 是否注册成功了
3. 返回值到底是资源对象，还是已经可直接挂进场景的运行时对象
4. 后续再次加载时为什么结果和第一次不一样

## 先看最小主链

先看一个最小例子：

```ts
await Laya.init({ designWidth: 1334, designHeight: 750 });

const logo = await Laya.loader.load("resources/ui/logo.png");
```

从运行时语义看，这一行代码不是“直接 new 出一个 `Texture`”。更接近下面这条主链：

1. `Laya.loader` 先根据 URL、扩展名和可选类型参数判断这是什么资源
2. 再找到处理这一类资源的 loader
3. 如果缓存里已有结果，直接返回缓存结果
4. 如果同一个资源正在加载，后续请求会并到同一条加载任务上
5. 如果需要实际下载，就先拿到原始内容
6. 再由对应 loader 把原始内容解析成引擎里的资源对象或可直接使用的结果对象

所以 `await Laya.loader.load(...)` 的重点不是“下载完成”，而是“这个 URL 已经按对应资源类型走完了解析流程”。

## URL 和类型是怎么决定后续流程的

`Laya.loader.load()` 并不是只看字符串路径本身。

最常见的两种写法是：

```ts
const tex = await Laya.loader.load("resources/ui/logo.png");
const tex2 = await Laya.loader.load("resources/ui/logo.png", Laya.Loader.IMAGE);
```

这里至少有两层判断：

1. 如果你显式给了类型，loader 会优先参考这个类型
2. 如果没显式给类型，引擎会根据扩展名去匹配已注册的资源处理规则

这也是为什么“路径写对了但结果不对”不一定是路径问题，也可能是：

1. 这个扩展名对应的 loader 没注册
2. 你传入了不合适的类型
3. 你以为自己要的是 `IMAGE`，但真正需要的是别的资源类型语义

一个特别容易混淆的例子是：`IMAGE` 和 `TEXTURE2D` 不是同一层概念。

在文档理解上，更稳妥的结论是：

1. `IMAGE` 更偏 2D 纹理加载入口语义
2. `TEXTURE2D` 更偏底层纹理资源语义
3. 你最终拿到什么对象，要看这一类资源的 loader 和解析规则

不要把“看起来都和图片有关”直接理解成“完全是一回事”。

## loader 选择依赖模块注册

`Laya.loader` 能处理哪些资源，不是只由 `Loader` 本体决定的。

对应能力通常要先由模块把自己的 loader 注册进运行时。

所以这类问题经常会出现：

```ts
const res = await Laya.loader.load("scenes/Main.ls");
```

代码形式没有问题，但如果对应模块没有完成注册，结果就可能不是你预期的那样可用。

这里真正要记住的是：

1. `Laya.loader` 是统一入口
2. 具体资源能不能被正确处理，取决于对应模块是否完成注册
3. “API 存在”不等于“这类资源的处理链已经接好”

这也是前面几篇文档提到“模块注册”边界的重要原因。

## 缓存和并发去重会改变你看到的行为

很多资源第一次加载和第二次加载的表现不同，不是错觉，而是资源系统本来就会先看缓存。

最常见的行为可以概括成三句：

1. 已缓存的资源，通常会直接返回缓存结果
2. 同一个资源如果正在加载，后续请求通常不会重复下载
3. 如果你显式忽略缓存，才会强制重新走加载流程

例如：

```ts
const a = Laya.loader.load("resources/ui/logo.png");
const b = Laya.loader.load("resources/ui/logo.png");

const [tex1, tex2] = await Promise.all([a, b]);
```

这里更接近“两个调用共享同一条资源加载任务”，而不是“各自重新下载一次”。

这会带来两个直接结果：

1. 你看到的并发加载次数，未必等于代码里调用次数
2. 后续资源释放、重新加载时，行为会受缓存状态影响

## `load()` 返回的不一定是最终运行时对象

最需要先建立的边界只有一句话：

`load()` 返回的是这类资源的加载结果，不一定是最终要加入显示树的运行时对象。

不同资源，返回值语义并不一样。

## 路径一：普通贴图更接近“直接可用结果”

例如：

```ts
await Laya.init({ designWidth: 1334, designHeight: 750 });

const tex = await Laya.loader.load("resources/ui/logo.png");

const sp = new Laya.Sprite();
sp.texture = tex;
Laya.stage.addChild(sp);
```

这条路径里，`load()` 返回的结果已经很接近你业务代码可以直接使用的对象。

主链可以简单理解成：

1. 识别这是图片相关资源
2. 下载原始图片数据
3. 解析成 `Texture` 这类运行时可直接消费的对象
4. 后续显示对象引用它进行渲染

所以对普通贴图来说，`load()` 之后通常离“可用”只差业务代码把它挂到某个运行时对象上。

## 路径二：图集加载返回的不只是“一张图”

图集的典型误区是把它理解成普通图片。

例如：

```ts
await Laya.loader.load("resources/ui/common.atlas");

const icon = Laya.loader.getRes("resources/ui/common/icon.png") as any;
```

这里真正发生的事更接近：

1. 先加载图集描述数据
2. 再关联图集大图资源
3. 再为图集中的小图建立可访问结果

所以图集的价值不是“返回一张大图”，而是“建立一组小图到运行时纹理结果的映射关系”。

这也是为什么图集加载完成后，你常常不是继续直接使用 `.atlas` 返回值本身，而是再按小图路径去取结果。

## 路径三：层级资源加载完成，不等于节点已经创建

对场景和预制体这类层级资源，最容易误解的一点是：

“加载完成”不等于“运行时对象树已经创建完成并挂进舞台”。

先看一个预制体例子：

```ts
await Laya.init({ designWidth: 1334, designHeight: 750 });

const prefab = await Laya.loader.load("resources/prefab/Hud.lh");
const hud = prefab.create();

Laya.stage.addChild(hud);
```

这里分成了两段：

1. `load("resources/prefab/Hud.lh")` 得到的是层级资源对应的资源对象
2. `prefab.create()` 才是把层级数据反序列化成真正的运行时对象

也就是说，层级资源的主链不是到 `load()` 就结束，而是：

1. 先加载层级数据本身
2. 收集它依赖的贴图、材质、子 prefab 等资源链接
3. 把这些依赖资源继续加载进来
4. 生成层级资源对应的资源对象
5. 在 `create()` 时再把层级数据解析成节点树，并绑定脚本、组件、引用关系

所以 prefab 的正确理解应该是：

1. `load()` 拿到的是“可实例化的资源对象”
2. `create()` 才拿到“实际运行时对象”

## `Laya.Scene.open(...)` 是层级资源链路的更高层封装

场景也属于层级资源，但业务代码更常直接写：

```ts
await Laya.Scene.open("scenes/Main.ls");
```

这比手写 `load() + create() + addChild()` 更高一层。

从运行时语义看，它背后仍然依赖类似的层级资源链路：

1. 先把 `.ls` 作为层级资源加载
2. 解析其依赖资源
3. 反序列化成场景对应的运行时对象树
4. 再执行场景实例级的打开流程

所以 `Laya.Scene.open(...)` 和 prefab `create()` 的差别，不是前者不走层级资源系统，而是前者把“加载、解析、创建、打开场景”串成了更完整的高层入口。

## 层级资源为什么还要单独走一次反序列化

因为层级资源文件保存的不是“已经活着的节点实例”，而是“描述如何构造节点树的数据”。

这类数据通常至少要回答几件事：

1. 节点类型是什么
2. 父子层级怎么组织
3. 每个节点挂了哪些组件或脚本
4. 属性值、资源引用和 prefab override 怎么恢复
5. 如果声明了 runtime，最终该实例化成哪个运行时类

所以层级资源真正进入运行时对象，依赖的是“按数据创建对象并恢复引用关系”的过程，而不是简单把 JSON 读出来就结束。

这也是为什么你会看到：

1. 有的层级资源加载成功，但 `create()` 时才暴露类型缺失问题
2. 脚本类或运行时类没有正确注册时，问题可能出现在实例化阶段
3. prefab 嵌套 prefab 时，最终结果依赖整个反序列化链是否完整

## 资源依赖会跟着层级资源一起进入运行时

对普通图片来说，依赖关系通常比较简单。

但层级资源不是只加载一个 `.ls` 或 `.lh` 文件就结束。

例如：

```ts
const prefab = await Laya.loader.load("resources/prefab/Enemy.lh") as Prefab;
```

它背后可能继续带出：

1. 贴图
2. 材质
3. 网格
4. 动画资源
5. 子 prefab

这也是“代码里只写了一个路径，实际却加载了很多文件”的原因。

对于层级资源，更准确的理解应该是：

1. 你显式请求的是入口资源
2. 运行时实际加载的是整个依赖链

## 资源释放为什么会影响后续行为

很多人第一次拿到资源后，会下意识认为它会一直常驻。

这不准确。

资源对象有自己的生命周期，后续是否仍在缓存中、依赖引用是否还在、是否已经被释放，都会影响下一次访问结果。

在理解层面，先记住三件事就够了：

1. 缓存命中时，后续 `load()` 可能直接返回已有结果
2. 某些资源会跟随引用计数和依赖关系参与生命周期管理
3. 资源被清理后，再次访问可能触发重新加载或恢复

例如：

```ts
const tex = await Laya.loader.load("resources/ui/logo.png");

Laya.loader.clearRes("resources/ui/logo.png");
```

这之后再次使用同一路径时，就不能再想当然地认为“内存里一定还是刚才那个对象状态”。

因此，很多“第二次打开场景不一样”“prefab 再创建时贴图丢了”“明明之前加载过为什么又请求一次”的问题，根因并不是加载 API 本身变了，而是资源生命周期已经进入了下一阶段。

## 最容易混淆的四件事

1. `load()` 完成，不等于最终运行时对象已经创建完成
2. 普通贴图、图集、层级资源的返回值语义不一样
3. 对层级资源来说，`create()` 和 `Laya.Scene.open(...)` 才是进入运行时对象阶段的关键一步
4. 模块注册、缓存状态和资源释放都会改变你后续看到的行为

## 结论

对 `LayaAir 3.4` 来说，理解 `Laya.loader.load()` 最重要的不是多记几种调用签名，而是先建立一条稳定主链：

1. 先识别 URL 和资源类型
2. 再选择对应 loader
3. 再处理缓存和并发去重
4. 再解析成资源对象或可直接使用的结果对象
5. 如果是层级资源，再继续反序列化成运行时对象树

把这条链路和 `Laya.Scene.open(...)`、prefab `create()`、资源释放边界分开之后，很多“明明 load 成功了，为什么还不能直接用”的问题就会清楚很多。
