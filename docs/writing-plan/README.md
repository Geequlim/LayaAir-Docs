# LayaAir 文档写作规划

## 目标

这份规划用于指导后续的 LayaAir 文档重写工作。

文档目标不是复刻官方目录，而是围绕“如何不依赖 IDE 界面，使用标准 Laya 项目与 `layaair` 命令行完成开发”来重组知识。

核心受众：

1. 普通技术开发者
2. 需要稳定检索和推理文档的 agent

核心约束：

1. 不以“点哪里、拖哪里”为主叙事。
2. 允许使用标准 Laya 项目、场景、预制体、资源、脚本、发布流程。
3. 优先解释项目结构、运行机制、数据关系、代码用法。
4. 官方文档中即使标题偏 IDE，也要抽取其中可复用的引擎知识。

## 说明

这份文档保留了偏细的主题拆分，适合作为选题池。

如果进入实际落地阶段，建议优先采用更收敛的站点大纲：

`docs/writing-plan/site-outline.md`

后者会把内容量不足的细项合并成更适合成篇的文档。

## 资料结论

当前可用参考资料分三层：

1. 官方中文文档仓库
2. LayaAir 引擎源码仓库
3. CLI / 插件相关的专项技能资料

建议的使用优先级：

1. 官方文档用于覆盖范围和使用场景盘点
2. 引擎源码用于校准概念、生命周期、资源机制、模块边界
3. skills 用于补 CLI、插件、agent 友好的写法模板

## 参考资料使用约束

`@references` 下的子项目只用于内部写作参考，不会作为最终文档站内容发布。

这意味着后续正式写作时应遵守这些规则：

1. 不在正文中引用本仓库里的 `references/...` 本地路径
2. 不把“去看 references 下某个文件”写进面向读者的文档
3. 可以提公开的开源项目、官方仓库或官方文档站，但不建议堆砌来源清单
4. 正式文档优先直接陈述整理后的知识，不要写成“摘录式”或“考据式”内容
5. 如果确实需要给读者补外部入口，优先给官方公开地址，而不是内部参考位置

### 官方文档的四类内容

1. 引擎核心功能
2. 可脱离 IDE 叙事重写的标准项目工作流
3. 明显的编辑器面板操作文档
4. 标题偏编辑器、正文却混有运行时原理的混合文档

后续重写时，必须按这个分类处理，不能直接照搬官方目录。

## 重写原则

1. 先讲对象模型，再讲操作路径。
2. 先讲运行时行为，再讲这些行为在项目中的落点。
3. 先给代码和数据结构，再补编辑器上下文。
4. 一篇文档只解决一个明确问题，避免官方文档那种主题混杂。
5. 每篇都尽量补齐：适用场景、前置条件、关键概念、步骤、边界条件、常见误区。
6. 能从源码确认的机制，优先用源码校准术语。
7. 无法脱离 IDE 的内容，不作为主线教程写，而作为补充说明或附录。

## 建议的新文档结构

不建议沿用官方“基础 / IDE / 2D / 3D / 发布”的原始拆法。

建议改成 8 个文档组，先建立开发心智模型，再进入各运行时子系统。

### 1. 入门与心智模型

目标：解释这套文档和官方文档的区别，以及“标准 Laya 项目但不依赖 IDE UI”到底意味着什么。

建议 5 篇：

1. LayaAir 文档阅读方式与目标边界
2. 标准 Laya 项目是什么
3. Laya 项目的关键目录与文件约定
4. 场景、节点、组件、脚本、资源的关系图
5. 选择源码、官方文档、CLI、skills 的阅读顺序

### 2. CLI 与项目工作流

目标：把“如何创建、运行、构建、验证项目”从 IDE 操作叙事中剥离出来。

建议 7 篇：

1. 安装与管理 `layaair` CLI
2. 使用 CLI 创建标准项目
3. 项目入口、启动场景与运行入口的关系
4. `assets`、`resources`、`src` 的职责划分
5. 预览、脚本执行与命令行调试
6. 构建平台、输出目录与常见参数
7. 资源校验与常见项目结构错误

### 3. 引擎运行时基础

目标：建立所有后续章节都会反复引用的核心机制。

建议 10 篇：

1. `Laya.init()`、`Config`、`LayaEnv` 与引擎启动流程
2. Stage、设计分辨率、缩放模式、屏幕模式
3. Node 树与显示对象层级
4. Component 与 Script 生命周期
5. 事件系统、输入与交互分发
6. Timer、Tween、帧循环与时间缩放
7. Loader 的资源类型、缓存与优先级
8. Resource 生命周期、引用关系与释放策略
9. 模块注册、`ModuleDef` 与按需能力装配
10. 调试入口、错误捕获与运行时观察点

### 4. 场景、预制体与序列化资源

目标：承接标准项目工作流，把 IDE 资产产物解释成代码可理解的对象。

建议 8 篇：

1. 场景文件、层级资源与运行时加载方式
2. 预制体是什么，适合解决什么问题
3. `.lh`、`.ls`、`.scene`、`.prefab` 等资源的角色区分
4. 预制体实例化、节点挂载与代码使用方式
5. 脚本绑定、`@regClass()` 与运行时类注册
6. 资源依赖、预加载与层级资源的引用收集
7. 场景切换、打开关闭与资源回收
8. 什么时候应该手写代码创建对象，而不是依赖序列化资源

### 5. 2D、UI 与界面组织

目标：把 2D 和 UI 的日常开发组织成代码优先的可检索知识。

建议 9 篇：

1. Sprite、Box、Label、Image 等常用 2D 节点
2. 文本、字体、位图字体与多语言注意事项
3. 布局、锚点、坐标系与相对定位
4. 贴图、图集、动态合批与动态图集
5. 动画、时间轴与常见 2D 效果组织方式
6. List、滚动容器与数据驱动 UI
7. 传统 UI 系统与 `ui2` 的区别
8. UI 预制体、局部复用与覆盖关系
9. 用代码组织 UI 与用资源组织 UI 的取舍

### 6. 3D 运行时

目标：围绕场景对象、渲染对象和资源对象搭建 3D 主线。

建议 11 篇：

1. Scene3D、Sprite3D、Transform 的基础关系
2. Camera、投影、层与可见性
3. Light、阴影、环境与天空盒
4. Mesh、材质、纹理与渲染器组件
5. PBR 材质与常见材质参数理解
6. 模型资源、glTF 与导入后的运行时使用
7. 动画、Animator、状态机与脚本协作
8. 粒子、拖尾、特效与可选模块
9. 后处理、屏幕后效与渲染扩展
10. Shader 与自定义渲染扩展的入门路径
11. 3D 性能观察点与常见资源问题

### 7. 物理、导航与交互能力

目标：整理那些官方散落在组件文档、编辑器文档和高级章节中的运行时能力。

建议 7 篇：

1. 2D 物理组件模型
2. 3D 物理组件模型
3. 碰撞、触发器与回调时机
4. 射线、拾取与交互检测
5. 路径导航与 NavMesh 的适用场景
6. 自定义物理或替换物理后端的边界
7. 物理与脚本生命周期协作注意事项

### 8. 发布、平台与扩展能力

目标：保留必要的平台与工程能力，但不让它们挤占主线。

建议 8 篇：

1. Web 构建与标准发布目录
2. 小游戏平台构建的共性与差异
3. 命令行构建脚本与自动化发布
4. 平台适配层与运行时差异的理解方式
5. 资源版本控制、包体与远程资源策略
6. 常见构建失败与资源丢失排查
7. IDE 插件与编辑器扩展能力总览
8. 什么时候需要阅读 editor plugin / build plugin 资料

## 建议篇数

完整第一版建议控制在 65 篇左右。

建议分层如下：

1. 主线必写：39 篇
2. 第二层增强：18 篇
3. 附录与扩展：8 篇

如果需要先做最小可用版本，建议先完成前 4 个文档组，共 30 篇左右。

### P0：先搭骨架

建议先写 6 篇，解决读者进入项目后的第一批问题：

1. 标准 Laya 项目是什么
2. 项目目录与关键文件约定

1. Component / Script 生命周期
2. 场景加载与切换
3. 预制体与代码实例化
4. `@regClass()` 与脚本绑定
5. 常用 2D 节点
6. UI 布局与列表
7. 2D 资源与图集
8. Scene3D / Sprite3D / Transform
9. Camera / Light / Material
10. 2D 物理基础
11. 3D 物理基础

### P2：扩展到完整能力面

补完 3D 进阶、渲染扩展、平台发布、导航、插件能力。

## 技能资料的利用方式

CLI / 插件相关的专项技能资料适合直接影响文档写法，但不适合作为完整内容骨架。

建议复用它的三个优点：

1. 命令行示例明确
2. 约束与坑点写得直接
3. 很适合 agent 检索

建议优先吸收进以下主题：

1. CLI 安装与版本管理
2. `create/build/run/validate` 工作流
3. 编辑器插件与 build plugin 的补充文档
4. 每篇文档中的“不要这样做”与“适用边界”小节

## 交接方式

后续实际写作建议按“文档组 -> 单篇卡片 -> 执行”推进。

推荐每篇文档在开写前先补一张写作卡片，至少包含：

1. 目标问题
2. 目标读者
3. 主要参考来源
4. 是否需要读源码校准
5. 是否需要 draw.io / Excalidraw / 关系表
6. 预计会拆成几篇

## 最终建议

这套项目的第一版，不应该追求“覆盖官方全部内容”，而应该优先做出一条清晰、可检索、能支撑真实开发的主线。

建议目标：

1. 先完成 30 篇左右的最小主线
2. 再扩展到 65 篇左右的完整第一版
3. 编辑器纯操作内容晚于运行时主线整理

如果后续开始正式写作，建议先从“入门与心智模型 + CLI 与项目工作流 + 引擎运行时基础 + 场景预制体”四组开始。
