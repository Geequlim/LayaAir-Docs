# P0 写作卡片

## 说明

这份文件用于把第一阶段的主线文章变成可直接分派的写作卡片。

约束默认继承当前规划：

1. 版本锁定 `LayaAir 3.4`
2. 少外链
3. 不写 IDE 操作过程
4. 示例少而精
5. 术语按源码语义统一

## 1. 什么是标准 Laya 项目

目标路径：`docs/getting-started/standard-project.md`

目标问题：

1. 一个标准 Laya 项目最少由哪些部分组成
2. 哪些目录和文件真的会影响开发、运行和发布

必须讲清：

1. `.laya` 是项目识别文件，不是运行入口
2. 开发的核心目录是 `src/` 和 `assets/`
3. `bin/` 不是常规资源根目录
4. 标准项目仍然适用于不依赖 IDE UI 的开发流程

建议结构：

1. 什么叫“标准项目”
2. 根目录里最重要的文件和目录
3. 哪些目录日常高频使用
4. 哪些目录知道作用即可
5. 对运行和发布真正有影响的配置点

建议示例：

1. 一个最小项目目录树
2. 一段非常短的推荐资源布局示例

主要内部参考类型：

1. 官方项目目录说明
2. 官方项目入口说明
3. 官方发布资源规则说明

源码锚点：

1. `Laya.ts`
2. 样例工程入口文件只作为反例说明，不作为标准项目规范

常见误区：

1. 把 `Main.ts` 当标准项目固定入口
2. 把 `bin/` 当资源根目录
3. 以为 `assets/` 下所有文件都会自动打进发布产物

## 2. 一个 Laya 项目是如何启动的

目标路径：`docs/getting-started/project-startup.md`

目标问题：

1. 引擎启动和项目启动有什么区别
2. 启动场景和启动脚本分别扮演什么角色

必须讲清：

1. `Laya.init()` 属于引擎启动
2. 启动场景 / 启动脚本属于项目入口层
3. 打开场景不是初始化引擎
4. 预览、运行、构建看到的“入口行为”可能不同

建议结构：

1. 启动分成哪两层
2. 引擎启动时做了什么
3. 项目入口有哪些常见形式
4. 预览和发布时入口差异在哪里

建议示例：

1. 一个最小 `Laya.init()` 示例
2. 一个最小场景打开示例

主要内部参考类型：

1. 官方项目入口说明
2. 官方脚本编译与入口说明
3. 官方运行配置说明

源码锚点：

1. `Laya.ts`
2. `Scene.ts`
3. `Config.ts`

常见误区：

1. 误以为当前场景预览一定会走启动脚本
2. 在 `Laya.init()` 之后再改初始化前配置
3. 拿 samples 的入口结构套用到标准项目

## 3. 如何用 `layaair` CLI 管理项目

目标路径：`docs/project-workflow/layaair-cli.md`

目标问题：

1. 不依赖 IDE UI 时，怎么完成项目创建、预览、构建和校验

必须讲清：

1. CLI 安装分 dispatcher 和 runtime 两层
2. 核心命令是 `create`、`run`、`build`、`validate`
3. 模板和平台名都应由 CLI 实时查询
4. `run --script` 适合执行注册类里的静态方法

建议结构：

1. 安装与版本管理
2. 创建项目
3. 预览项目
4. 构建项目
5. 校验资源
6. 执行脚本

建议示例：

1. 每个核心命令只保留 1 个最小示例
2. 保留一段最短脚本运行示例

主要内部参考类型：

1. CLI 技能资料
2. 旧命令行资料只用于历史对照，不直接沿用

源码锚点：

1. 这里以 CLI 行为资料为主，不强求引擎源码锚点

常见误区：

1. 安装完 dispatcher 就以为 CLI 已可用
2. 不先列模板/平台就硬写名字
3. 把旧的 IDE 命令行流程当成 `layaair` CLI

## 4. 项目里的资源应该放在哪里

目标路径：`docs/project-workflow/asset-layout.md`

目标问题：

1. 资源目录应该怎么划分，才能同时兼顾开发、加载和发布

必须讲清：

1. 开发期资源总根是 `assets/`
2. `resources` 更像强制打包目录，不是代码加载资源的必需目录
3. 场景依赖资源和代码动态资源的发布规则不同
4. 不是“放在 assets 里就一定进包”

建议结构：

1. 先区分三类问题：管理、加载、发布
2. 场景资源应该怎么放
3. 动态加载资源应该怎么放
4. 哪些例外需要单独配置
5. 推荐目录约定

建议示例：

1. 推荐的资源目录树
2. 一个场景引用资源示例
3. 一个 `Laya.loader.load("resources/...")` 示例

主要内部参考类型：

1. 官方项目目录说明
2. 官方发布资源规则说明
3. 图集资源规则说明

源码锚点：

1. `Loader.ts`
2. `URL.ts`
3. `AssetDb.ts`

常见误区：

1. 动态资源没放进 `resources`
2. 在 `bin/` 放普通项目资源
3. 混用旧的 `res/...` 和现在的 `resources/...`

## 5. 引擎启动后，运行时里有哪些关键入口

目标路径：`docs/runtime-foundation/runtime-entry-points.md`

目标问题：

1. `Laya.init()` 之后，开发者真正会持续接触哪些运行时入口

必须讲清：

1. `Laya.stage`
2. `Laya.loader`
3. `Laya.timer` / `systemTimer` / `physicsTimer`
4. `Config` 和 `LayaEnv` 的影响范围
5. 模块注册为什么会影响功能是否可用

建议结构：

1. `Laya.init()` 做了什么
2. 全局入口对象总览
3. 初始化前后哪些配置点最重要
4. 为什么有时“代码写对了但功能还是不可用”

建议示例：

1. 一个最小初始化示例
2. 一段访问 `Laya.stage` / `Laya.loader` 的极短示例

主要内部参考类型：

1. 生命周期相关官方资料
2. 运行时系统相关源码

源码锚点：

1. `Laya.ts`
2. `Config.ts`
3. `LayaEnv.ts`
4. `ModuleDef.ts`

常见误区：

1. 在初始化后才改必须预设的配置
2. 以为全局 `Laya` 对象可访问就代表所有模块都已注册
3. 不理解 `onAwake`、`onEnable`、`onStart` 的边界

## 6. 资源加载之后会经历什么

目标路径：`docs/runtime-foundation/resource-loading.md`

目标问题：

1. `Laya.loader.load()` 之后，到“真正可用”之间究竟发生了什么

必须讲清：

1. URL / 类型识别
2. loader 选择
3. 缓存与并发去重
4. 解析成运行时对象或资源对象
5. 对层级资源再反序列化成对象树
6. 资源依赖和释放规则

建议结构：

1. 从 load 调用到结果对象的主链
2. 贴图、图集、层级资源三条典型路径
3. prefab 为什么“加载完成”不等于“节点已创建”
4. 资源缓存和释放为什么会影响后续行为

建议示例：

1. 一个图片加载示例
2. 一个图集/小图访问示例
3. 一个 prefab `create()` 示例

主要内部参考类型：

1. 2D / 3D 资源加载相关官方资料
2. 层级资源与资源系统源码

源码锚点：

1. `Loader.ts`
2. `HierarchyLoader.ts`
3. `HierarchyParser.ts`
4. `PrefabImpl.ts`
5. `Resource.ts`

常见误区：

1. 没有导入对应模块导致 loader 没注册
2. 把 `IMAGE` 和 `TEXTURE2D` 当成同一件事
3. 误以为 prefab 资源加载完成就已经实例化
4. 误以为资源会一直常驻不会被释放
