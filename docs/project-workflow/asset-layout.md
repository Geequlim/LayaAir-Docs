# 项目里的资源应该放在哪里

资源放在哪里，表面上是目录命名问题，实际上同时影响三件事：

1. 开发时怎么管理
2. 运行时怎么加载
3. 发布时哪些资源会进入产物

如果这三件事没有分开看，就很容易得到两个错误结论：

1. 只要放在 `assets/` 里，发布后就一定可用
2. 只要代码里能写出路径，资源就一定会被正确收集

更稳妥的理解方式是：`assets/` 是开发期资源总根，`resources/` 更像强制打包的便利目录，而场景依赖资源、配置包含资源和代码动态资源的进入产物规则并不完全一样。

## 先分清三类问题

讨论资源目录前，先把问题拆成三类：

### 1. 管理问题

资源文件在项目里放哪里，团队最容易维护。

### 2. 加载问题

运行时是由场景、预制体自动带出资源，还是由代码主动调用 `Laya.loader.load()`。

### 3. 发布问题

最终构建时，哪些资源会被收进产物，哪些不会。

同一个文件是否“应该放这里”，必须同时回答这三类问题。

## `assets/` 是开发期资源总根

在标准项目里，开发期资源主目录是 `assets/`。

通常这些内容都会放在这里：

1. 场景
2. 预制体
3. 图片与图集
4. 音频
5. 材质与模型
6. 配置文件

例如：

```text
assets/
  scenes/
    Boot.ls
    Main.ls
  prefab/
    Enemy.lh
  textures/
    env/
      bg.png
  audio/
    hit.wav
  resources/
    ui/
      hud.atlas
      hud.png
```

这里先记住一个边界：`assets/` 解决的是开发期资源组织问题，不等于“这里的全部文件都会自动发布”。

## 场景依赖资源应该怎么放

如果一个资源主要是被场景、预制体或层级资源引用的，它首先应该按内容类型和使用关系组织，而不是强行都塞进 `resources/`。

例如：

```text
assets/
  scenes/
    Main.ls
  prefab/
    Enemy.lh
  textures/
    enemy/
      body.png
  materials/
    enemy.lmat
```

这类资源的典型特征是：

1. 你不会在业务代码里频繁手写它的路径
2. 它更常通过场景文件、预制体文件或材质引用关系被带出来
3. 资源之间存在层级依赖

如果 `Main.ls` 里已经引用了某个预制体，预制体又引用了贴图和材质，那么运行时打开这个场景时，会沿着层级依赖继续解析这些资源。

一个极简例子：

```ts
await Laya.Scene.open("scenes/Main.ls");
```

这里代码只写了场景路径，但真正进入运行时的通常不止一个 `.ls` 文件，还包括这个场景依赖的其他资源。

所以，场景资源的核心不是“都放进 `resources/`”，而是“让引用关系清晰、目录边界稳定”。

## `resources/` 不是代码加载资源的必需目录

资源能不能被代码加载，关键不在于它是否位于 `resources/`，而在于：

1. 它最终是否进入了发布产物
2. 运行时路径是否可访问

把资源放进 `assets/resources/`，主要是为了更直接地确保它会被打进包，而不是因为代码只能从这里加载。

例如：

```text
assets/
  resources/
    ui/
      logo.png
    audio/
      click.wav
    config/
      stages.json
```

对应的运行时路径通常写成：

```ts
await Laya.loader.load("resources/ui/logo.png");
await Laya.loader.load("resources/config/stages.json");
```

这里的目录语义要分两层看：

1. 文件在项目里实际位于 `assets/resources/...`
2. 运行时代码可以通过 `resources/...` 路径访问它

所以 `resources/` 的价值更接近：

1. 快速确保一批资源会进入产物
2. 适合少量需要兜底打包的动态资源
3. 不适合作为整个项目的默认资源收纳区

## 不要把 `resources/` 当成默认方案

`resources/` 很重要，但它不是“唯一正确目录”。

如果所有资源都塞进 `assets/resources/`，通常会带来几个问题：

1. 场景依赖资源和纯动态资源混在一起
2. 目录结构失去业务含义，只剩“能加载”这一个标准
3. 后续排查冗余资源、场景依赖和加载边界会变得更难

更好的约定通常是：

1. 场景、预制体、材质、模型等按资源关系组织
2. 只有那些你明确希望以“强制打包”方式兜底的资源，再谨慎放进 `resources/`

可以把它理解成：`resources/` 是“强制打包资源区”，不是整个项目的唯一资源区。

## 发布时，场景依赖资源和动态资源的规则不同

资源是否进入构建产物，不能只看它是不是放在 `assets/` 下。

更接近实际的判断方式是：

1. 被场景、预制体等层级资源引用到的资源，会跟随依赖链参与收集
2. 代码需要动态加载的资源，必须先确保它能进入最终产物
3. 没有被引用、也不在有效动态加载规则内的文件，不能默认认为一定会进入产物

所以这两种资源的思路并不一样：

1. 场景资源靠依赖关系进入运行时与产物
2. 动态资源靠“进入产物 + 路径可访问”进入运行时

这也是为什么“只要在 `assets/` 里就行”这个理解不够准确。

## 哪些资源需要单独留意

有几类资源最容易因为目录放置混乱而出问题。

### 图集和小图

如果你加载的是图集，实际使用时拿到的可能不只是单个图片文件，而是图集描述和其中的小图资源关系。

这类资源更适合整组放在同一块目录里，例如：

```text
assets/resources/ui/
  hud.atlas
  hud.png
```

### 配置数据

如果 JSON、文本或二进制配置需要由代码显式读取，可以放进 `resources/` 作为强制打包手段，但这不是唯一方案。

```ts
const stages = await Laya.loader.load("resources/config/stages.json");
```

### 临时测试资源

临时资源如果直接散落在 `assets/` 根下，后续很容易分不清它到底属于场景依赖、动态资源，还是已经废弃。

与其放成这样：

```text
assets/
  a.png
  test.wav
  data1.json
```

不如尽早归类到明确目录。

## `bin/` 不是资源源目录

不要把普通项目资源长期放进 `bin/`。

`bin/` 更接近预览期或中间产物目录。你可能会在里面看到运行页面和部分输出文件，但它不是标准项目的资源管理起点。

如果某个资源只存在于 `bin/`，而不在 `assets/` 的正式结构里，通常说明你的项目目录已经偏离标准工作流了。

## 推荐的目录约定

如果你需要一个能同时兼顾开发、加载和发布的简单约定，可以从下面开始：

```text
assets/
  scenes/
    Boot.ls
    Main.ls
  prefab/
    ui/
      Hud.lh
    gameplay/
      Enemy.lh
  textures/
    gameplay/
      enemy.png
  materials/
    gameplay/
      enemy.lmat
  audio/
    gameplay/
      hit.wav
  resources/
    ui/
      common.atlas
      common.png
    audio/
      click.wav
    config/
      stages.json
```

这个约定的重点不是目录名本身，而是三条边界清楚：

1. `assets/` 是开发期总根
2. 场景依赖资源按引用关系和内容类型组织
3. `resources/` 适合少量需要强制打包的资源

## 常见误区

### 误区 1：只有放进 `resources/` 代码才能加载

不是。代码能否加载资源，取决于资源是否进入产物，以及运行时路径是否可访问。

### 误区 2：把 `bin/` 当常规资源目录

这会把开发源文件和预览产物混在一起。

### 误区 3：把 `resources/` 当成默认兜底目录

这会快速解决“打不进去包”的短期问题，但也会让资源依赖边界越来越糊。

### 误区 4：混用旧的 `res/...` 和现在的 `resources/...`

在 `LayaAir 3.4` 项目里，应优先使用当前项目约定的 `resources/...` 路径语义，不要把旧项目习惯直接照搬进来。

### 误区 5：以为 `assets/` 下所有文件都会自动进入发布包

不是。是否进入产物，还要看依赖关系和动态加载规则。

## 结论

对 `LayaAir 3.4` 来说，最实用的资源目录原则可以收成四句话：

1. 开发期资源总根是 `assets/`
2. 场景依赖资源按引用关系组织，不必强行都放进 `resources/`
3. `resources/` 更适合少量强制打包资源，不适合作为默认策略
4. 不是“放在 `assets/` 里”就一定会进入最终产物

只要先把管理、加载、发布这三条线分开，资源目录应该怎么放，通常就不会再混乱。
