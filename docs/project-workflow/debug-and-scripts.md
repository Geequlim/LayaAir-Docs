# 开发时如何调试和执行脚本

开发阶段常见的两类工作，其实不是一回事：

1. 调试项目运行时
2. 执行工程脚本

前者是让项目跑起来，观察启动链、场景、资源和运行时行为；后者是借助 CLI 跑一段注册过的 TypeScript 静态方法，完成导出、校验、批处理这类工程任务。

## 先分清两种 `run`

`layaair run` 有两种常见用法：

1. 不带 `--script`：启动本地预览服务
2. 带 `--script`：执行注册脚本

最小示例：

```bash
layaair run -p .
```

```bash
layaair run -p . --script=MyTools.run
```

虽然命令都叫 `run`，但语义完全不同。不要把“预览项目”和“执行工程脚本”混成同一件事。

## 调试项目运行时

日常开发里，最常用的入口还是先把项目预览起来：

```bash
layaair run -p .
```

这会启动 CLI 内置的本地预览服务。它的作用是把项目跑进浏览器运行时，方便你检查：

1. `Laya.init()` 是否真的完成
2. 是否进入了预期的启动脚本或首个场景
3. 资源、场景脚本、计时器和输入事件是否按预期工作

如果项目是“能启动但行为不对”，优先按启动链排查，而不是先怀疑构建流程：

1. 初始化前配置是否放在 `Laya.init()` 之前
2. `await Laya.init(...)` 是否完成
3. 是否真的进入了首个场景或首个代码入口
4. 入口之后依赖的资源是否成功加载

## 预览环境里放什么调试代码

预览环境和正式运行环境不一定完全一样，所以调试代码最好明确限定作用范围。

最短写法：

```ts
await Laya.init({ designWidth: 1334, designHeight: 750 });

if (LayaEnv.isPreview) {
  Laya.Stat.show();
  console.log("preview mode");
}
```

这种写法适合放：

1. 性能面板
2. 额外日志
3. 仅预览时启用的测试入口

不适合放：

1. 正式运行也必须依赖的初始化逻辑
2. 会改变首页链路的临时分支，但发布前忘记清理

核心原则只有一条：调试分支可以依赖 `LayaEnv.isPreview`，正式启动链不要依赖它才能成立。

## 浏览器里优先看什么

当项目已经通过 `layaair run` 跑起来后，最有价值的通常不是继续改命令，而是直接看浏览器开发者工具。

优先检查这几类信息：

1. `Console`：启动报错、资源报错、脚本注册失败
2. `Network`：场景、图片、配置文件是否成功返回
3. `Sources`：断点是否真的打在当前执行文件上

如果你怀疑入口没走到，最短的确认方式往往就是在启动链上加一条日志：

```ts
await Laya.init({ designWidth: 1334, designHeight: 750 });
console.log("init done");
await Laya.Scene.open("scenes/Main.ls");
```

这类日志应该尽量打在边界点上，而不是到处散落。

## 执行工程脚本

另一类开发任务不是调试画面，而是让 CLI 帮你跑一段工程代码。

典型场景包括：

1. 导出数据
2. 批量处理资源
3. 生成配置文件
4. 做一次独立校验

这时用的是 `run --script`：

```ts
@IEditorEnv.regClass()
export class MyTools {
  static async run(output: string): Promise<void> {
    console.log(output);
  }
}
```

```bash
layaair run -p . --script=MyTools.run --script-args="dist/data.json"
```

这里最重要的是三点：

1. `--script` 指向 `类名.静态方法名`
2. 目标类需要先注册
3. `--script-args` 传的是一整段参数字符串

如果你写成实例方法，或者类没有注册，CLI 就不能按预期调用它。

## 临时脚本适合一次性任务

有些任务你不想正式放进项目里，只想临时执行一次，这时可以额外传入脚本文件：

```bash
layaair run -p . --script=AX.test --script-file=/tmp/a.ts
```

这更适合：

1. 一次性排查
2. 临时数据转换
3. 短期验证某段工具逻辑

它只影响当前这次执行，不等于把这个文件正式纳入项目工程。

## 一个顺手的开发循环

对 `LayaAir 3.4` 来说，开发时比较顺手的命令行节奏通常是：

1. `layaair run -p .`，先把项目预览起来
2. 修改启动链、场景脚本或资源后，再次观察预览结果
3. 需要工程处理时，插入一次 `layaair run -p . --script=...`
4. 资源改动较多时，再补一次 `layaair validate ... -p .`

这条链路的重点不是命令越多越好，而是把三件事分开：

1. 预览
2. 脚本执行
3. 资源校验

## 常见误区

1. 把 `layaair run` 默认理解成“执行工程脚本”
2. 把预览环境里临时加的调试分支，当成正式启动逻辑的一部分
3. 入口异常时，不先检查 `Laya.init()` 和首个场景是否真的执行
4. 用 `run --script` 去调用实例方法
5. 临时脚本跑通后，误以为它已经自动变成项目正式资源

## 结论

开发阶段如果只先抓住一条主线，可以记成这样：

1. 用 `layaair run -p .` 调试项目运行时
2. 用 `LayaEnv.isPreview` 隔离预览专用调试逻辑
3. 用 `layaair run --script` 执行注册过的工程脚本

把这三件事分清后，启动问题、调试代码和工程脚本就不容易再互相干扰。
