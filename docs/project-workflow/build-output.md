# 项目构建后会得到什么

构建不是让项目“再运行一次”，而是把开发期工程整理成目标平台可以直接交付的产物。

从代码、工程和部署角度看，构建后最值得关心的不是“目录里多了哪些文件”，而是下面四件事：

1. 入口文件变成了什么
2. 代码以什么形式被执行
3. 资源最终按什么路径进入产物
4. 这批文件应该怎样被整体部署

## 先把构建产物理解成一个可交付目录

无论目标是 Web 还是小游戏平台，构建结果都可以先理解成一组可交付文件，而不是开发目录的镜像。

它通常至少包含三层内容：

1. 平台入口层
2. 运行时代码层
3. 资源与配置层

也就是说，构建后的目录重点不是保留你原来的 `src/`、`assets/` 形状，而是把它们整理成目标平台真正能启动、能加载、能发布的文件集合。

## 从代码角度看，构建后发生了什么

对代码来说，构建最核心的变化有两件事：

1. 开发期源码不会按原样直接作为交付物使用
2. 运行时真正执行的是构建后的脚本入口和资源路径

可以把关系先粗略理解成：

```text
开发期：src/ + assets/
构建后：入口文件 + 运行脚本 + 可访问资源
```

例如，你在项目里写的是：

```ts
await Laya.Scene.open("scenes/Boot.ls");
```

构建后真正重要的，不是 `Boot.ls` 在源码里放在哪里，而是：

1. 这条路径对应的场景文件是否进入了产物
2. 它依赖的预制体、贴图、音频是否也被一起收集
3. 运行时访问路径是否仍然成立

所以构建成功，不等于运行一定正常。只要入口脚本还在，但场景依赖资源没有正确进入产物，项目仍然可能在正式环境里启动失败。

## 从工程角度看，构建后会得到哪些类型的文件

不要把构建产物理解成源码目录的备份。更准确地说，它是对工程的一次发布整理。

常见会得到下面几类文件：

1. 启动入口文件
2. 编译后的脚本文件
3. 场景、预制体、图集、音频、配置等资源文件
4. 平台需要的额外配置文件

一个 Web 产物通常可以粗略理解成这样：

```text
build/
  index.html
  js/
  resources/
  other assets...
```

这里不要把目录名当成死规则，重点是结构关系：

1. 浏览器需要一个入口页面
2. 页面会再加载脚本和资源
3. 资源路径要和运行时代码中的访问方式对应起来

小游戏平台的产物虽然长得不完全一样，但本质仍然是同一件事：

```text
build/
  平台入口脚本
  平台配置文件
  运行时代码
  资源文件
```

区别主要不在“有没有资源”，而在“谁来启动这些代码”。

Web 由浏览器页面启动，小游戏平台则由各自的平台入口和平台配置启动。

## Web 构建产物该怎么理解

如果目标是 `web`，最实用的理解方式是：构建结果就是一份静态站点。

它至少要满足三件事：

1. Web 服务器能把入口页面返回出来
2. 入口页面引用的脚本文件可访问
3. 脚本继续请求的场景和资源文件也可访问

从部署角度看，这意味着你交付的不是某个单独的 `.js` 文件，而是一整个目录。

例如，下面这种关系才是完整交付单元：

```text
index.html
js/main.js
resources/scenes/Boot.ls
resources/ui/common.png
```

如果只上传其中一部分，例如只传脚本不传资源，或者只传页面不传依赖脚本，构建产物就不完整。

## 小游戏平台构建有哪些共性

不同小游戏平台的细节不一样，但从工程视角看，共性通常比差异更重要。

可以先抓住四点：

1. 都需要平台自己的启动入口
2. 都需要平台识别的配置文件
3. 都要把运行时代码和资源一起交付
4. 都不能把开发期源码目录直接当成上传目录

所以小游戏构建产物更像“平台可识别的应用包”，而不只是一个普通静态目录。

从部署动作看，也通常不是“扔到任意 Web 根目录就结束”，而是要把整包交给对应平台工具或平台上传流程。

## 哪些源码内容不会按原样保留

构建后，最容易产生误解的是“我在项目里看到的目录，发布后也会原样存在”。

实际不应该这样理解。

更稳妥的判断是：

1. `src/` 是开发期代码来源，不是最终交付目录名承诺
2. `assets/` 是开发期资源来源，不等于产物里一定还叫 `assets/`
3. `bin/` 更接近预览期或中间产物，不应反向当成正式工程源目录

也就是说，源码目录的职责是方便开发，产物目录的职责是方便目标平台运行。两者相关，但不是一一照抄。

## 对部署真正有意义的，是“整包关系”

从部署角度看，构建结果最重要的特征是文件之间的相对关系已经被确定。

你真正要交付的是：

1. 可以启动的入口
2. 入口依赖的脚本
3. 脚本依赖的资源
4. 平台要求的配置文件

所以部署时最常见的正确动作是：

1. Web：把整个构建目录作为静态站点发布
2. 小游戏平台：把整个平台构建目录作为上传包提交

而不是从里面手工挑几个“看起来重要”的文件单独复制。

## 常见误区

### 误区 1：构建后得到的就是源码拷贝

不是。构建结果是面向运行和交付的目录，不是开发目录快照。

### 误区 2：预览能跑，正式产物就一定能跑

不是。正式环境是否正常，还要看入口文件、资源收集和产物路径是否成立。

### 误区 3：只要有主脚本文件，其他文件可以后补

通常不行。场景、图集、贴图、音频、配置文件都可能是启动链的一部分。

### 误区 4：构建目录适合长期手工维护

不适合。构建目录更应该被视为可重复生成的交付结果，而不是日常编辑源。

## 出问题时先查哪一层

如果构建完成后运行异常，通常先按这三层排查：

1. 入口层：入口文件或平台配置是否正确
2. 代码层：构建后的脚本是否正常启动
3. 资源层：场景、预制体和动态加载资源是否真的进入产物

很多“构建成功但打不开”的问题，最后都不是引擎初始化本身出错，而是产物里的依赖关系断了。

## 结论

项目构建后，得到的本质上不是一份源码副本，而是一份面向目标平台的可交付包。

对 LayaAir 3.4 来说，最实用的理解方式是：

1. Web 产物是一整份静态运行目录
2. 小游戏平台产物是一整份平台可识别的应用包
3. 代码是否能跑，取决于入口、脚本和资源是否一起被正确整理进去
4. 部署时应整体交付构建目录，而不是只挑单个文件
