# 资源版本和远程资源该怎么规划

资源版本和远程资源，表面上像部署问题，实际会同时影响四件事：

1. 首包有多大
2. 更新后旧缓存怎么失效
3. 资源路径边界是否稳定
4. 线上问题能不能快速定位

如果这四件事没有分开看，就很容易走向两个极端：

1. 所有资源都打进本地包，结果包体越来越重
2. 所有资源都改成远程，结果路径、缓存和回退都开始失控

更实用的做法不是先问“要不要上 CDN”，而是先把本地包资源、远程资源、版本控制和路径边界拆开。

## 先分清两类资源

最先要分清的，不是目录名，而是交付方式。

### 本地包资源

这里说的本地包资源，是指会跟随当前构建产物一起发布的资源。

对 Web 来说，它们通常和页面、脚本一起组成一整份静态目录。

对小游戏平台来说，它们通常属于首包，或者属于平台可识别的本地分包。

这类资源的特点通常是：

1. 启动链依赖它
2. 缺了就会直接影响首屏或主流程
3. 希望离线、弱网或首次安装时也尽量可用

常见例子：

1. 启动场景
2. 登录前后的基础 UI 图集
3. 核心音效
4. 首次进入就会用到的配置

### 远程资源

远程资源是指不随当前构建产物一起交付，而是在运行时再从远端地址获取的资源。

这类资源的特点通常是：

1. 不一定参与首屏
2. 更新频率可能更高
3. 体积可能更大
4. 允许按需下载

常见例子：

1. 大章节资源
2. 活动资源
3. 热更新配置
4. 高体积音视频

先把这个边界立住，后面讨论版本和缓存才不会混乱。

## 版本控制真正想解决什么

资源版本控制的目标，不是“每次发布都带一个版本号”这么简单。

更实际的目标通常只有三个：

1. 新版本上线后，运行时不要继续误用旧缓存
2. 没变化的资源，尽量继续复用缓存
3. 当线上资源和代码不匹配时，能快速知道错在谁

所以资源版本控制本质上是在解决“同一路径的内容变了以后，缓存该怎么处理”。

如果没有这个意识，最常见的问题就是：

1. 代码已经发新版了，用户仍然拿到旧图集
2. 场景文件更新了，但依赖贴图还在走旧缓存
3. 活动资源已替换，但部分用户仍在使用旧文件

## 缓存失效要按资源粒度思考，不要按发版口号思考

“这次发版版本号加一”并不能自动解决缓存问题。

真正有效的思路是：

1. 内容变了，就让这个资源的可访问地址发生变化，或者让它的版本映射发生变化
2. 内容没变，就尽量保持原地址或原版本映射，继续吃缓存

实践上常见是两种做法：

1. 文件名或 URL 带内容版本
2. 维持稳定路径，再通过 `fileconfig.json` 这类版本映射表改写真实访问地址

可以把它简单理解成：

```text
逻辑路径 -> 版本映射 -> 最终请求地址
```

例如业务代码里仍然写：

```ts
await Laya.loader.load("resources/ui/common.atlas");
```

但线上真正请求的可能已经被映射成：

```text
resources/ui/common.atlas?v=20260624_01
```

或者：

```text
resources/ui/common.8f3a2d.atlas
```

这里的重点不是具体长什么样，而是同一个逻辑路径在内容变化后，最终请求地址必须可区分。

## `fileconfig` 的职责边界

`fileconfig.json` 更适合被理解成“资源版本映射表”，而不是“万能资源管理中心”。

它更适合解决的是：

1. 业务代码仍保持稳定逻辑路径
2. 构建或发布阶段把逻辑路径映射到带版本的真实地址
3. 运行时加载前先套用这层映射

它不适合替代下面这些职责：

1. 资源是否该进首包
2. 某资源是否应该改为远程
3. 整个项目的目录组织方式

更直接地说：

1. `assets/` 解决开发期资源组织
2. 构建产物决定哪些文件实际被交付
3. `fileconfig` 解决已交付资源的版本映射问题

不要把三者混成一件事。

## `base path` 的职责边界

`base path` 更适合解决“同一套逻辑路径，默认从哪里取资源”。

例如：

```ts
Laya.URL.basePath = "https://cdn.example.com/game/";
await Laya.loader.load("resources/ui/logo.png");
```

这里更接近：

```text
resources/ui/logo.png
-> https://cdn.example.com/game/resources/ui/logo.png
```

所以 `base path` 的主要职责是“路径根切换”，不是“版本控制”。

它适合用在：

1. 整批资源迁到 CDN
2. 某个包前缀统一走远端
3. 测试环境和正式环境切换资源源站

它不适合单独承担：

1. 哪些资源该失效
2. 哪个文件是新版本
3. 首包和远程包的更新协调

一个更稳妥的理解方式是：

```text
逻辑路径边界：我在代码里怎么写
base path 边界：默认去哪里找
version/fileconfig 边界：找到的是哪一版
```

## 包、`fileconfig`、`base path` 三条线不要混

实际项目里最容易乱的，就是把“包边界”“路径边界”“版本边界”写在同一套心智里。

更清楚的分法是：

### 1. 包边界

解决“哪些资源跟首包走，哪些资源延后下载”。

这是一条交付边界。

### 2. `fileconfig` 边界

解决“同一个逻辑路径当前该落到哪个真实文件”。

这是一条版本映射边界。

### 3. `base path` 边界

解决“这批逻辑路径最终应该从本地目录还是远端域名取”。

这是一条资源源站边界。

如果三条线分不清，最常见的结果就是：

1. 一改 CDN 地址就顺手改业务代码路径
2. 一做版本号就把整个目录名全改掉
3. 一做远程资源就不知道哪些文件还在首包里

## 本地包资源该怎么选

默认先把下面这些放在本地包里，通常更稳妥：

1. 引擎启动后立刻需要的场景和依赖资源
2. 首屏 UI 的基础图集和字体
3. 登录、创角、主流程刚进入就要用到的配置
4. 弱网下仍希望尽量可用的核心资源

一个简单例子：

```text
首包
  Boot.ls
  Main.ls
  基础 UI 图集
  登录配置
```

这里的判断标准不是“它小不小”，而是“缺了以后首流程还能不能成立”。

## 远程资源什么时候值得上

远程资源不是越早上越好。只有在下面几类场景里，它的复杂度才通常值得：

1. 包体压力已经成为真实约束
2. 资源体积很大，而且用户不会一次性全部用到
3. 活动或章节更新频繁，明显独立于主程序版本
4. 多平台首包限制明显，必须拆首包
5. 你已经有稳定的 CDN、回源、缓存和监控能力

如果还处在项目早期，资源量不大，发布频率也不高，优先把本地包结构和资源引用边界理顺，通常比提前远程化更划算。

## 哪些资源特别适合远程

下面几类资源通常更适合考虑远程化：

1. 大型章节包
2. 限时活动资源
3. 可独立替换的公告图、运营图
4. 高分辨率可选资源
5. 不影响首流程的补充语音和视频

例如：

```text
本地包
  resources/ui/common.atlas
  scenes/Boot.ls

远程
  events/summer-2026/*
  chapters/chapter-07/*
  audio/voice-pack-a/*
```

这类拆法的重点不是目录名，而是业务上能不能独立下载、独立失败、独立回退。

## 远程资源最容易踩的坑

### 远程了，但路径边界没稳定

如果业务代码里到处直接拼完整 CDN URL，后面切域名、切目录、切灰度源都会很痛苦。

更稳妥的是让业务代码尽量保留稳定逻辑路径，把源站切换收敛到 `base path` 或更集中的配置层。

### 远程了，但版本策略还是整包思维

远程资源最怕“整批统一改大版本号”。

这样虽然粗暴可用，但会让没变的资源也全部失去缓存价值。

更实用的方向是按文件或按资源组做版本失效。

### 远程了，但没有失败回路

远程资源一定要先接受一件事：下载可能失败、超时、被缓存污染，或者 CDN 还没完全刷新。

所以至少要想清楚：

1. 加载失败后是重试、降级还是跳过
2. 关键配置失败后是否有本地兜底
3. 运营资源更新失败时是否允许回退上一版

### 远程了，但仍把启动链放在远端

如果启动场景、首屏图集、首个关键配置都在远端，首屏稳定性会直接受网络影响。

除非业务明确接受这种代价，否则不建议这么做。

## 一个更实用的规划顺序

如果你正在给项目定资源策略，比起先讨论工具细节，更建议先按下面顺序做决定：

1. 先列出启动链必需资源，固定为本地包
2. 再列出体积大但非首流程必需的资源，作为候选远程资源
3. 为逻辑路径建立稳定前缀，不要让业务代码直接依赖部署域名
4. 再决定版本方案是文件名改写、查询串，还是 `fileconfig` 映射
5. 最后才讨论 CDN、回源和缓存时间配置

这条顺序的好处是：先稳定资源边界，再上版本和远程，不容易反过来让部署策略绑架项目结构。

## 一个够用的最小策略

如果项目还不大，可以先从下面这套最小策略开始：

1. 核心启动资源全部随构建产物发布
2. 代码里统一使用稳定逻辑路径，例如 `resources/...`、`chapters/...`
3. 用 `fileconfig` 或等价版本映射处理变更文件的缓存失效
4. 只把章节、活动、大体积可选资源放到远端
5. 不要把整站所有资源一开始就做成远程

例如：

```text
逻辑路径
  resources/ui/common.atlas
  resources/config/bootstrap.json
  chapters/chapter-07/scene.ls

部署策略
  resources/...        -> 本地包或首包
  chapters/...         -> 远程资源

版本策略
  fileconfig / 文件名改写 只作用于已变更文件
```

## 常见误区

### 误区 1：资源版本就是每次发版统一改一个总版本号

这只能解决一部分问题，但通常会过度失效缓存。

### 误区 2：只要用了 CDN，资源管理就算做完了

不是。CDN 解决的是分发效率，不自动解决首包边界、版本映射和失败回路。

### 误区 3：远程资源一定比本地包高级

不是。它只是更适合一部分资源，不是默认更优。

### 误区 4：`base path` 就等于版本控制

不是。`base path` 更像资源源站切换；版本控制要另外处理缓存失效问题。

### 误区 5：`fileconfig` 能顺便决定资源该不该进包

不是。资源是否进包，首先是构建和交付策略问题。

## 结论

对 LayaAir 项目来说，资源版本和远程资源最实用的规划原则可以收成五句话：

1. 先分清本地包资源和远程资源，不要把所有文件都按同一规则处理
2. 版本控制的核心目标是让变更资源失效缓存，而不是给发版动作贴标签
3. `fileconfig` 负责版本映射，`base path` 负责资源源站，包策略负责交付边界
4. 启动链和核心体验资源优先留在本地包
5. 只有当包体、更新频率和运营形态真的需要时，远程资源的复杂度才值得引入
