# 什么时候需要看插件和构建扩展资料

多数项目从创建到上线，靠 `layaair` CLI 的标准子命令就够了：`create`、`run`、`validate`、`build`。插件和构建扩展并不是默认需要的东西，它们只在标准工作流解决不了某个具体问题时才变得相关。

所以更实用的判断方式不是“要不要学插件”，而是先问：当前这条 CLI 主线是不是已经卡住了？

## 先分清两件不同的事

容易被混在一起的，是“编辑器插件”和“构建扩展”。它们的定位不一样：

1. 编辑器插件：在编辑阶段介入，通常用来增强编辑器内的资源处理、面板、导入导出、自定义流程。
2. 构建扩展：在构建阶段介入，用来在 `layaair build` 之外额外改写、补充或生成构建产物里的内容。

可以简单理解成：

```text
编辑期   <- 编辑器插件介入
构建期   <- 构建扩展介入
运行期   <- 不由插件负责
```

也就是说，插件和扩展影响的是“项目怎么被编辑和怎么被打包”，而不是“游戏运行时本身多出了什么能力”。

## 标准 CLI 工作流什么时候够用

如果你的需求落在下面这些范围内，通常不需要碰插件和扩展：

1. 创建项目、预览、校验资源、按平台构建
2. 跑一段项目内已注册的工程脚本，例如导出数据、批处理资源
3. 场景、预制体、图集、音频按常规依赖链进入产物
4. 不同平台只是入口形式不同，资源收集链本身稳定

这些场景下，`layaair run`、`layaair build`、`layaair validate` 和 `layaair run --script` 已经覆盖了日常工程动作。这时候去看插件文档，通常只是增加心智负担，不会带来实际收益。

## 什么时候插件和扩展才值得看

只有当标准 CLI 主线明确无法满足某个具体需求时，插件和扩展文档才值得读。常见的触发信号有几类：

### 1. 需要在构建阶段改动产物内容

标准 `build` 产出的是一份可交付目录，它的内部关系由构建流程决定。如果你需要：

- 在产物里额外生成配置、清单或索引文件
- 在构建后自动改写某些资源的引用路径
- 把外部数据源的结果合并进构建产物

这类需求已经超出“执行一次构建”的语义，构建扩展才是对口的工具。

### 2. 需要自定义资源导入或处理

如果某些资源的导入规则、格式约定或命名规范和默认行为不一致，并且这种差异需要在编辑阶段反复发生，靠手动调整不可持续，那么编辑器插件更适合承担这部分职责。

典型例子：

- 特定格式的地图、关卡、配置在导入时需要额外解析
- 资源命名需要按团队约定自动归类或重命名
- 第三方工具产出的中间文件需要被转换成项目可用的资源

### 3. 需要把团队工程流程沉淀进编辑器

当团队需要稳定的、可复用的工程流程，而不是每次靠临时脚本或手动操作时，插件的价值才开始显现。

例如：

- 一键执行某套资源校验和修复流程
- 把自定义的导出、打包、上报步骤固化进编辑器
- 为特定项目类型提供专属面板或操作入口

这些场景的共同点是：它们是反复发生的工程动作，值得被沉淀，而不是一次性任务。

## 怎么判断扩展文档值不值得读

在决定是否深入插件和扩展文档前，可以先问自己三个问题：

1. 这个需求能不能用 `layaair run --script` 配合一段工程脚本解决？
2. 这个需求是不是只在构建后改产物，而不涉及编辑期行为？
3. 这个需求是一次性的，还是会长期反复出现？

对应的判断大致是：

```text
能靠 --script 解决          -> 不需要插件
只涉及构建产物改写          -> 优先看构建扩展
涉及编辑期资源或流程         -> 优先看编辑器插件
一次性需求                  -> 用临时脚本，不必上插件
长期反复出现的工程流程       -> 值得沉淀成插件或扩展
```

这套判断的核心是：插件和扩展是为了减少重复工程成本，而不是为了“让流程看起来更完整”。

## 一个常见的误判

最容易出现的误判是：一遇到构建或资源问题，就先去翻插件和扩展文档，怀疑是不是需要自定义流程。

实际上，多数构建异常的根源仍然是三条链没成立：

1. 入口链
2. 依赖链
3. 资源链

只有当这三条链在标准工作流下已经稳定，而你仍然有一个明确的、标准流程无法覆盖的需求时，插件和扩展才真正进入有效讨论范围。

否则更稳妥的做法是：先把 `layaair build -l`、`layaair validate`、`layaair run --script` 用好，把工程边界理清，再判断是否还需要进一步扩展。

## 最小判断流程

如果用一个最短路径来决定是否需要看插件和构建扩展资料：

1. 先确认需求是否能落在 `create`、`run`、`build`、`validate` 的标准范围内
2. 如果不行，再看能不能用 `run --script` 配一段工程脚本解决
3. 如果仍然不行，区分需求属于编辑期还是构建期
4. 属于构建期的，看构建扩展
5. 属于编辑期资源或流程的，看编辑器插件

只有走到第 4 或第 5 步，插件和扩展文档才真正值得投入时间。

## 常见误区

### 误区 1：插件是项目的标配

不是。大多数项目从创建到发布，标准 CLI 工作流就足够。插件是为了解决标准流程覆盖不到的具体问题，不是项目成熟度的标志。

### 误区 2：构建扩展可以替代标准构建

不能。构建扩展是在标准构建基础上追加或改写，而不是另起一套构建链。如果标准 `build` 的产物关系本身不稳定，扩展只会让问题更难定位。

### 误区 3：遇到构建问题先查插件文档

不建议。构建问题的高频原因仍然是入口、依赖和资源链没成立。先把标准工作流和产物结构理清，再判断是否真的需要扩展。

### 误区 4：一次性任务也要做成插件

不值得。一次性的工程动作用 `run --script` 配临时脚本更划算，插件和扩展更适合长期反复出现的流程。

## 结论

插件和构建扩展不是日常开发的必读内容，而是当标准 CLI 工作流明确无法覆盖某个具体需求时的下一步选择。

更实用的态度是：

1. 先把 `create`、`run`、`build`、`validate` 这条主线用熟
2. 再把 `run --script` 当成工程脚本出口
3. 只有当需求确实落在编辑期资源处理或构建产物改写，而且会反复出现时，再去读插件和构建扩展资料
