# 多平台发布时最容易踩哪些坑

多平台发布最容易让人误判的地方，不是“某个平台配置少填了一项”，而是把预览、构建、运行、资源收集这几件事混成了一件事。

无论目标是 Web 还是小游戏平台，真正高频的坑通常都落在三层：

1. 项目层：工程结构和构建习惯本身不稳定。
2. 运行时层：代码在不同发布目标下依赖了不稳定行为。
3. 资源层：资源路径、依赖关系和进入产物的规则没有理清。

如果这三层没分开看，常见结果就是：本地预览正常，构建也成功，但换个目标平台后启动失败、白屏、缺图、缺脚本或动态资源打不开。

## 先把多平台发布理解成“同一项目的多种交付形式”

Web 和小游戏平台看起来差异很大，但从工程视角看，核心问题是一致的：

1. 谁来启动你的运行时代码。
2. 运行时代码按什么路径找资源。
3. 哪些文件最终真的进入交付产物。

也就是说，多平台发布不是把同一套源码“原样复制”给不同平台，而是把同一项目整理成多种可运行交付包。

所以真正该优先追的，不是某个平台按钮怎么点，而是这三条链是否一直成立：

1. 入口链成立。
2. 依赖链成立。
3. 资源链成立。

## 项目层最常见的坑

## 1. 把预览成功当成发布成功

`layaair run -p .` 适合确认项目能否启动，但它不是正式发布结果。

最常见的误判是：

1. 本地预览能开场景。
2. 于是默认构建后也一定能开。
3. 最后上线时才发现入口资源或动态资源没有进入产物。

更稳的顺序是：

1. 先用 `run` 看启动链有没有明显问题。
2. 再用 `build -l` 和 `build <platform> -p .` 看真实目标产物。
3. 最后按构建后的入口、脚本和资源关系排查。

## 2. 把构建目录当长期手工维护目录

构建产物是可重复生成的交付结果，不是开发源目录。

如果你在构建目录里手改脚本、手拷资源、手补配置，短期可能能跑，后面很容易出现两类问题：

1. 下一次重新构建把手工改动覆盖掉。
2. 同一个问题只在某个构建包里被“临时修好”，源码里其实没解决。

多平台发布时这个问题会更明显，因为你往往不只生成一个目标目录。

## 3. 平台名、CLI 版本和构建命令靠记忆写

跨平台构建最忌讳“沿用旧项目习惯”。

更稳的做法是：

1. 先 `layaair build -l`，确认当前 CLI 真正支持的平台名。
2. 在脚本或 CI 里尽量写明 `--version`。
3. 不要把旧教程里的平台参数直接照搬过来。

这类问题表面像“命令行问题”，本质上仍然是工程交付链不稳定。

## 运行时层最常见的坑

## 4. 依赖了只在某个目标下凑巧成立的入口行为

Web 由页面入口启动，小游戏平台由各自平台入口和配置启动。

这意味着你不能把“当前预览入口碰巧怎么工作”理解成所有目标都一样。

真正要确认的是：

1. 启动代码是不是只依赖标准初始化链。
2. 首场景是不是通过稳定路径打开。
3. 启动时依赖的配置、资源和脚本是不是都进入了产物。

一个短例子：

```ts
await Laya.init({ designWidth: 1334, designHeight: 750 });
await Laya.Scene.open("scenes/Boot.ls");
```

这段代码本身很简单，但它成立的前提至少还有三件事：

1. `Boot.ls` 进入了产物。
2. `Boot.ls` 依赖的资源链完整。
3. 运行时路径在目标平台上仍然成立。

## 5. 代码里写对了路径，却没确认资源有没有被收进包

这是 Web 和小游戏平台都非常常见的坑。

例如：

```ts
await Laya.loader.load("resources/config/stages.json");
```

这句代码能否在发布后成立，不只取决于路径字符串有没有写对，还取决于：

1. 这个文件最终是否进入产物。
2. 构建后的运行时路径是否还能访问到它。
3. 它是不是被你放进了稳定的动态资源组织方式里。

很多“某平台读不到配置”的问题，最后不是 loader 有问题，而是文件根本没被带进去。

## 6. 混淆“加载成功”和“运行时对象已可用”

对普通图片、图集、场景、预制体，这几类资源的加载语义并不一样。

例如：

```ts
const prefab = await Laya.loader.load("resources/prefab/Hud.lh");
const hud = prefab.create();
```

这里 `load()` 之后拿到的还不是最终显示对象，真正进入运行时对象阶段还要看 `create()`。

多平台发布时，如果你只看到“资源加载 Promise 成功了”，就很容易漏掉后面的实例化、脚本绑定、依赖恢复问题。

## 7. 依赖注册、脚本类或资源类型在发布后丢失

有些问题不是路径错，而是运行时处理链没有完整接上。

典型表现：

1. 场景能打开，但脚本不生效。
2. 预制体能加载，但 `create()` 时出错。
3. 某类资源在开发期可用，发布后解析结果不对。

这类问题排查时，不要只盯着资源文件本身，还要反查：

1. 对应脚本类在发布后的代码里是否真的存在。
2. 相关模块和处理链是否在运行时可用。
3. 问题出在下载阶段，还是出在解析和实例化阶段。

## 资源层最常见的坑

## 8. 以为 `assets/` 下所有文件都会自动进入发布包

不是。

更接近实际的规则是：

1. 场景、预制体依赖资源会沿依赖链参与收集。
2. 代码动态加载资源要先确保能进入产物。
3. 没被引用、也没进入有效动态加载规则的文件，不能默认认为会被发布。

这也是为什么“本地项目里明明有这个文件”不能证明“发布后一定有这个文件”。

## 9. 把 `resources/` 当成万能兜底目录

`resources/` 很有用，但它更适合少量明确要强制打包的动态资源。

如果把所有资源都塞进去，短期会减少“打不进去”的焦虑，长期通常会带来三类问题：

1. 场景依赖资源和动态资源混在一起。
2. 资源边界变糊，后面很难查谁该常驻、谁该拆包。
3. 不同平台出问题时，难以判断到底是依赖链问题还是动态加载组织问题。

更稳的做法是：

1. 场景资源按引用关系组织。
2. 动态加载资源再谨慎放进 `resources/`。
3. 不要用一个大兜底目录掩盖资源管理本身的混乱。

## 10. 混用旧路径习惯和当前路径语义

跨平台时，路径问题最怕“旧项目里这么写也能跑”。

例如把旧的 `res/...` 习惯继续带进当前项目，往往会让下面三件事一起变乱：

1. 资源实际放在哪里。
2. 代码应该怎么加载。
3. 发布后路径应该长什么样。

多平台发布时，路径语义必须先统一，不然不同目标上的异常会变得非常碎。

## 11. 图集按文件类型合，而不是按一起出现的界面合

这类问题在 Web 和小游戏平台上都会变成发布和运行问题，而不只是美术整理问题。

例如把所有按钮、所有图标、所有小图都打进一个超级 `common.atlas`，常见后果是：

1. 某个页面只用一小部分图，也得把整大组资源带上。
2. 一处小改动会影响整张图集。
3. 后续很难判断哪些资源应该跟着哪个页面一起加载或释放。

更稳的思路仍然是按界面和生命周期拆：登录一组、大厅一组、战斗一组、公共小组件一组。

## 12. 临时资源、测试资源长期散落在正式目录里

跨平台发布时，最容易在最后阶段暴露出来的，不一定是缺资源，也可能是冗余资源和错误引用。

例如：

1. 场景还挂着测试贴图。
2. 配置还指向临时 JSON。
3. 图集里混入已经废弃的资源。

这些问题在单一预览流程里可能不明显，一旦要做多目标构建和回归，就会迅速放大排查成本。

## 出问题时先按哪条链排查

如果构建成功但某个平台运行异常，通常先不要直接猜“平台兼容性有问题”。

更稳的排查顺序是：

1. 入口链：目标产物的入口和启动配置是否正确。
2. 代码链：初始化和首场景打开是否真的执行到。
3. 依赖链：场景、预制体、脚本类、模块注册是否完整。
4. 资源链：动态资源和层级依赖资源是否真的进入产物。
5. 路径链：构建后的访问路径是否仍和代码约定一致。

这个顺序的价值在于：先查文件和链路是否存在，再查更细的运行时表现。

## 一份更稳的发布前检查表

在 Web 和小游戏平台之间切目标前，可以先快速过一遍：

1. 是否先用 `layaair build -l` 确认了平台名。
2. 是否明确当前使用的 CLI 版本。
3. 首场景路径是否稳定，且不依赖预览期特例。
4. 动态加载资源是否明确进入产物。
5. 场景和预制体依赖是否已经通过真实资源链收集。
6. 是否混用了旧路径语义。
7. 是否有人在构建目录里做了手工补丁。
8. UI 图集是否按界面和生命周期拆分，而不是无限堆进 `common`。

## 结论

多平台发布最容易踩的坑，归根到底不是“平台太多”，而是把工程、运行时、资源三层边界混掉了。

对当前项目工作流来说，最实用的结论是：

1. 先把构建产物当成完整交付包，而不是源码镜像。
2. 先把 `run`、`build`、`validate` 的职责分开。
3. 先把场景依赖资源和动态加载资源分开管理。
4. 先确认路径、依赖和资源收集链在目标产物里真的成立。

只要这几条边界稳定，多平台差异通常会缩小到入口形式不同，而不会演变成整条发布链反复失控。
