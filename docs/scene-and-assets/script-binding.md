# 场景和预制体里的脚本是怎么绑定的

很多人第一次看到场景或预制体里的脚本丢失，会把下面几件事当成一件事：

1. 资源有没有加载成功
2. 节点树有没有创建出来
3. 脚本类有没有被运行时认出来
4. 场景里的脚本绑定和代码里的 `addComponent()` 是不是同一条路径

但从运行时角度看，这其实是两条不同链路。

## 先分清两条绑定路径

最容易混淆的，是“代码直接挂脚本”和“场景数据恢复脚本”。

### 路径一：代码直接添加组件

```ts
const { regClass } = Laya;

@regClass()
class Rotator extends Laya.Script {
  onUpdate(): void {
    this.owner.rotation += 1;
  }
}

const sprite = new Laya.Sprite();
Laya.stage.addChild(sprite);
sprite.addComponent(Rotator);
```

这条路径里，运行时已经直接拿到了 `Rotator` 这个类本身。

更接近：

1. 业务代码明确指定要加哪个类
2. 引擎直接创建这个组件实例
3. 再把它挂到当前节点上

这里的关键是“类引用就在代码里”，不是“按名字去查类”。

### 路径二：场景或预制体按序列化数据恢复脚本

```ts
await Laya.Scene.open("scenes/Main.ls");
```

或者：

```ts
const prefab = await Laya.loader.load("resources/prefab/Enemy.lh") as Laya.Prefab;
const enemy = prefab.create();
```

这时运行时面对的不是“业务代码已经把类传进来了”，而是“层级数据里记录了某个脚本或 `runtime` 类型，需要在反序列化时把它还原出来”。

所以这里的关键变成了：

1. 资源先被加载
2. 层级数据开始反序列化
3. 运行时根据序列化信息查找对应类
4. 找到后再创建脚本实例或运行时实例
5. 再恢复属性、引用和节点关系

也就是说，场景和预制体里的脚本绑定，本质上属于“按数据恢复类”的问题。

## `@regClass()` 解决的到底是什么

`@regClass()` 最容易被误解成“只要写脚本就必须加，不然类不能运行”。

更准确的说法是：

`@regClass()` 主要解决“让运行时在反序列化时认出这个类”。

最小例子：

```ts
const { regClass } = Laya;

@regClass()
class Player extends Laya.Script {}
```

当场景、预制体或某个节点的 `runtime` 数据里记录了 `Player` 这个类型时，运行时需要从已注册的类表里把它找回来。

如果不注册，常见结果不是“资源打不开”，而是：

1. `.ls` 或 `.lh` 仍然可能加载成功
2. 节点树也仍然可能部分创建成功
3. 但脚本组件恢复失败，或 `runtime` 替换失败

所以 `@regClass()` 的重点不是继承关系，而是运行时类注册。

## 运行时类注册为什么是硬边界

场景和预制体文件保存的是序列化数据，不是已经活着的对象实例。

这些数据至少要描述：

1. 节点类型是什么
2. 节点上挂了哪些组件和脚本
3. 某个节点有没有声明 `runtime`
4. 组件字段和引用关系该怎么恢复

运行时在还原这些数据时，必须把“数据里的类型标记”重新对应到“当前代码里的类”。

因此类注册不是可选优化，而是序列化恢复链的一部分。

可以把它理解成：

1. `addComponent(SomeScript)` 是代码直接拿类构造器
2. 场景或 prefab 恢复脚本，是运行时先查类表再决定创建哪个类

两条路径最终都会得到脚本实例，但拿到类的方式不同。

## 脚本绑定和 `runtime` 恢复，本质上是同一类问题

很多人只把类注册和“脚本组件”联系起来，其实节点 `runtime` 也是同一条边界。

例如某个节点在序列化数据里声明了运行时类，运行时恢复时同样要做这件事：

1. 读取这个节点声明的目标类
2. 在已注册类中找到它
3. 用这个类替换默认节点类型或补充运行时行为

所以脚本丢失和 `runtime` 失效，经常不是两个独立问题，而是同一个根因：

运行时没能把序列化类型正确还原成真实类。

## 为什么“注册了”还可能绑定失败

只记 `@regClass()` 还不够。

从运行时结果看，至少还要同时满足两件事：

1. 这个类已经完成注册
2. 发布后的运行时代码里，这个类真的存在

最容易出问题的情况是：

1. 类只写在某个脚本文件里
2. 场景数据引用了它
3. 但入口代码从来没有让这个文件进入最终运行时

这时即使你在源码里写了 `@regClass()`，运行时也还是可能找不到它，因为那段代码根本没有进入当前执行环境。

所以更稳妥的理解是：

1. `@regClass()` 解决“已加载到运行时的类如何登记”
2. 代码可用性解决“这个类本身有没有真的被带进来”

两者缺一个都可能导致场景或 prefab 绑定失败。

## 场景和预制体里的绑定，通常在什么时候发生

对场景来说，常见入口是：

```ts
await Laya.Scene.open("scenes/Main.ls");
```

对预制体来说，常见入口是：

```ts
const prefab = await Laya.loader.load("resources/prefab/Enemy.lh") as Laya.Prefab;
const enemy = prefab.create();
```

两者的共同点是：

1. 先加载层级资源
2. 再进入反序列化创建对象树
3. 在创建过程中恢复脚本、组件、`runtime` 和引用关系

所以很多问题会表现成：

1. `load()` 成功了
2. `Laya.Scene.open()` 或 `prefab.create()` 之后对象也出来了
3. 但行为不对，脚本方法没执行

这时就不能只盯着“资源是否下载成功”，还要看类型恢复链是否完整。

## 绑定失败最常见的原因

可以优先按下面几类排查。

### 1. 类没有注册

例如：

```ts
class EnemyAI extends Laya.Script {}
```

如果它要被场景或 prefab 数据恢复，通常应注册成运行时可识别的类。

### 2. 类注册了，但脚本代码没有进入当前运行时

这类问题常见于：

1. 类只存在于源码文件里
2. 运行入口没有实际用到这段文件
3. 最终执行环境里没有这段类定义

结果就是类表里也不会有它。

### 3. 资源加载成功，但失败发生在实例化阶段

场景和 prefab 的脚本绑定，不一定在 `load()` 阶段暴露问题。

更常见的是：

1. 资源对象已经拿到了
2. 但在 `Laya.Scene.open()` 或 `prefab.create()` 的反序列化阶段才发现类缺失

所以“资源能 load 成功”不等于“脚本一定能恢复成功”。

### 4. 绑定成功了，但脚本看起来像没运行

这时问题可能不在类注册，而在节点和组件状态。

例如：

1. 宿主节点没有进入有效层级
2. 节点 `active = false`
3. 组件 `enabled = false`

这种情况更接近“脚本实例存在，但没进入有效生命周期”，不是“绑定失败”。

### 5. `runtime` 和脚本类混着看，导致误判

有时节点树出来了，但自定义运行时类没有生效；有时 `runtime` 恢复了，但脚本组件缺失。

这两者都属于“按数据恢复类”的链路问题，但恢复目标不是同一个类。

排查时要分清：

1. 是脚本组件没恢复
2. 还是节点 `runtime` 没替换成功

## 为什么直接 `addComponent()` 和序列化恢复的结果看起来不同

很多人会拿这两段代码做对比：

```ts
node.addComponent(PlayerCtrl);
```

```ts
const scene = await Laya.Scene.open("scenes/Main.ls");
```

它们最终都可能让某个节点拥有 `PlayerCtrl`，但语义不同：

1. `addComponent()` 是当前代码立刻决定加什么
2. 序列化恢复是层级数据先决定“应该是什么”，运行时再去还原

这会带来几个直接差别：

1. `addComponent()` 只依赖当前代码里拿得到类
2. 序列化恢复还依赖类注册和类型查找
3. 序列化恢复还会同时恢复序列化保存的字段值和引用关系

所以场景和 prefab 里的脚本，不只是“晚一点调用了一次 `addComponent()`”，而是走了一整条反序列化恢复链。

## 一个最小判断方法

遇到场景或 prefab 里的脚本不生效，可以先把问题分成三问：

1. 这是绑定失败，还是绑定成功但没进入激活链
2. 这个类有没有完成运行时注册
3. 当前运行时代码里，这个类本身是否真的存在

如果这三问先分清，大多数问题都会快很多。

## 结论

场景和预制体里的脚本绑定，关键不是“脚本文件写没写”，而是下面这条链是否完整：

1. 层级资源被正确加载
2. 反序列化阶段开始按数据恢复对象树
3. 脚本类和 `runtime` 类能被运行时查到
4. 对应类已经注册，且代码本身真实存在
5. 恢复出的脚本实例再进入正常的节点和组件生命周期

把这条链和“代码直接 `addComponent()`”分开之后，再看场景、prefab、脚本丢失和 `runtime` 失效问题时，会清楚很多。
