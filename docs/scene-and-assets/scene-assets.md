# 场景资源该怎么预加载和回收

很多人第一次处理场景资源时，会把下面几件事当成同一件事：

1. 场景打开前先把资源准备好。
2. 场景关闭后把画面关掉。
3. 场景用过的资源从内存里消失。

这三件事有关，但不是一条边界。

如果不先拆开看，就很容易出现两类误判：

1. 以为 `scene.close()` 之后资源一定已经释放。
2. 以为只要调用过 `Laya.loader.load()`，后面就不需要再管资源生命周期。

更稳妥的理解方式是：

1. 场景资源先沿依赖链加载。
2. 已加载资源通常会进入缓存，后续场景可复用。
3. 场景实例关闭、销毁，和资源缓存清理，不是同一步。

## 先看场景资源真正包含什么

一个场景资源不是只有一个 `.ls` 文件。

例如：

```ts
await Laya.Scene.open("scenes/Main.ls");
```

代码里只写了一个场景路径，但运行时通常会继续处理它依赖的内容，例如：

1. 场景里的图片和图集。
2. prefab。
3. 材质、模型、网格。
4. 音频。
5. 子层级资源。

所以对场景资源来说，更准确的理解不是“加载一个场景文件”，而是“加载一个场景入口和它的依赖链”。

## 预加载解决的是“先把依赖链准备好”

如果你直接这样写：

```ts
await Laya.Scene.open("scenes/Main.ls");
```

那 `open()` 会把“加载场景资源”和“打开场景实例”串在一起。

而预加载要解决的问题是：

1. 用户还没进入这个场景时，先把资源准备好。
2. 等真正打开场景时，减少等待时间。
3. 把加载阶段和打开阶段拆开。

最小写法可以是：

```ts
await Laya.loader.load("scenes/Main.ls");
await Laya.Scene.open("scenes/Main.ls");
```

这里可以先这样理解：

1. `load()` 先把场景资源及其依赖链准备好。
2. `open()` 再把这份已可用的场景资源变成真正的场景实例。

## 为什么预加载场景时，依赖资源也会一起进来

因为场景、prefab 这类层级资源保存的不是孤立文件路径，而是整个层级引用关系。

例如一个 `Main.ls` 里引用了：

1. `Hud.lh`
2. `enemy.lmat`
3. `common.atlas`

那么预加载 `Main.ls` 时，运行时通常不会只记住这个 `.ls`，而是继续把这些依赖资源也准备好。

所以场景预加载真正有价值的地方，不是少写几行代码，而是：

1. 你只声明入口资源。
2. 运行时沿依赖链把一整组资源准备好。

## 什么时候只预加载场景入口就够了

如果一个场景的大部分资源都已经通过场景、prefab、材质引用关系串好了，那么通常只预加载场景入口就够了。

例如：

```ts
await Laya.loader.load("scenes/Battle.ls");
```

这类场景的特征通常是：

1. 主要资源都挂在场景层级里。
2. 业务代码不会临时再拼很多额外资源路径。
3. 场景依赖关系稳定。

这时更适合把重点放在资源组织和引用关系是否清晰，而不是手工维护一大串额外加载列表。

## 什么时候要补充手工预加载

并不是所有资源都会被场景依赖链自动带出来。

最常见的例外是：资源路径只存在于业务代码里，而不在场景数据里。

例如：

1. 根据关卡配置动态决定加载哪张图。
2. 根据角色选择动态加载不同皮肤。
3. 某个弹窗运行时才按字符串路径加载图集。
4. 音频、配置表、远程资源不直接挂在当前场景层级上。

这时就要单独预加载：

```ts
await Promise.all([
  Laya.loader.load("scenes/Battle.ls"),
  Laya.loader.load("resources/config/levels.json"),
  Laya.loader.load("resources/audio/boss-enter.wav")
]);
```

可以先用一句话判断：

场景数据里看不到的动态资源，通常就不能指望场景预加载自动兜底。

## 缓存和复用才是第二次打开更快的原因

很多人看到第二次打开同一场景更快，会误以为“引擎记住了上次那个场景实例”。

通常不是。

更接近真实的情况是：

1. 上一次打开后，相关资源还在缓存里。
2. 这一次打开时，不必把整条依赖链重新加载一遍。
3. 运行时主要重做的是场景实例化和打开流程。

例如：

```ts
await Laya.Scene.open("scenes/Main.ls");
await Laya.Scene.open("scenes/Main.ls");
```

两次 `open()` 拿到的不是同一个场景实例。

更准确的理解应该是：

1. 场景实例通常是新的。
2. 底层资源可能是复用缓存结果。

## `close()` 处理的是“场景实例退出打开状态”

先看最容易混淆的一句：

```ts
scene.close();
```

它解决的核心问题是：

1. 这个场景不再作为当前打开场景运行。
2. 场景实例退出当前运行时打开状态。

它不等于：

1. 所有节点都被彻底销毁。
2. 场景依赖资源立刻从缓存里清空。

所以 `close()` 的边界更接近“关场景”，不是“清资源”。

## `destroy()` 处理的是“这个实例不要再用了”

如果你手里持有一个场景实例，真正不可逆的边界是销毁。

例如：

```ts
const scene = await Laya.Scene.open("scenes/Main.ls");
scene.destroy();
```

这里要分清：

1. `destroy()` 针对的是这个场景实例和它的节点树。
2. 它表示这份实例生命周期结束，不应该再继续复用。
3. 它仍然不等于“相关资源缓存必然已经全部清掉”。

所以：

1. `close()` 更偏场景打开状态。
2. `destroy()` 更偏实例生命周期终点。
3. 资源要不要继续留在缓存，是另一条线。

## `clearRes()` 处理的是“把资源缓存清掉”

资源清理更接近下面这类操作：

```ts
Laya.loader.clearRes("scenes/Main.ls");
```

这里要先建立一个最重要的边界：

`clearRes()` 处理的是资源缓存，不是场景实例树。

也就是说，它关注的是：

1. 某个资源入口及相关缓存结果要不要继续留着。
2. 后续再次访问时，是否需要重新加载。

它不负责：

1. 帮你自动关闭当前正在运行的场景。
2. 帮你把还在使用这些资源的实例改成可用状态。

因此，更安全的顺序通常是：

1. 先让场景实例退出运行。
2. 确认没有别的对象继续依赖这些资源。
3. 再清对应资源缓存。

## 为什么不能一关场景就立刻 `clearRes()`

因为资源是否还能被别的对象使用，不能只看“这个场景关没关”。

例如下面几种情况都很常见：

1. 两个界面共用同一套 UI 图集。
2. 大厅和战斗场景共用一批公共音频。
3. 某个 prefab 已经从场景里取出来，仍在别处继续使用。
4. 你做了对象池，回收的是节点实例，不是资源文件本身。

这时如果场景一关就直接清资源，后续很容易出现：

1. 再次使用时重新加载抖动。
2. 仍在使用的对象资源丢失。
3. 难以定位的显示或播放异常。

## 一条更稳的切场景写法

最简单的切场景思路通常是：

```ts
let currentScene: Laya.Scene | null = null;

await Laya.loader.load("scenes/Battle.ls");

currentScene?.close();

currentScene = await Laya.Scene.open("scenes/Battle.ls");
```

这条链路重点在于：

1. 先预加载新场景。
2. 再结束旧场景的打开状态。
3. 最后打开新场景。

这样做的价值是把“加载等待”尽量放到切换前面，而不是让用户在切场景瞬间一起承受。

## 什么时候适合保留缓存复用

下面这些情况，通常更适合让资源继续留在缓存里：

1. 场景会频繁来回切换。
2. 公共 UI 图集、字体、通用音频会长期复用。
3. 短时间内还会再次打开同一场景。
4. 重新加载成本明显高于保留内存成本。

例如大厅和背包反复切换、主界面弹窗反复打开关闭，这类内容通常更适合优先复用，而不是每次都清。

## 什么时候需要手工做资源回收

真正需要手工资源管理的，通常不是“小项目所有场景都要手工 clear”，而是下面这些更明确的情况：

1. 某个场景体量大，占用内存明显高。
2. 这个场景短时间内不会再回来。
3. 某批资源只属于一次性流程，比如章节结算、首包剧情、教学关。
4. 设备内存压力明显，自动缓存复用已经不划算。
5. 动态加载了很多不在公共依赖里的资源。

可以先记一个判断标准：

如果你已经很明确地知道“这批资源后面不会立刻再用”，手工回收才更有价值。

## 手工回收时，先分三层再决定动作

做资源回收时，最好先把对象分成三层：

1. 场景实例层：当前还在不在运行。
2. 节点实例层：有没有被别的地方拿去继续复用。
3. 资源缓存层：底层贴图、图集、材质、音频是否还要保留。

只有这三层都看过，才不容易误删。

一个最小例子可以写成：

```ts
const scene = await Laya.Scene.open("scenes/Boss.ls");

scene.close();
scene.destroy();
Laya.loader.clearRes("scenes/Boss.ls");
```

这里更适合表达的语义是：

1. 不再运行这个场景。
2. 不再保留这份场景实例。
3. 连它的入口资源缓存也一起清掉。

这类写法更适合一次性大场景，不适合高频来回切换的界面。

## 不需要把所有资源管理都改成手工模式

很多项目一开始就想“严格手工管理所有场景资源”，结果很快把逻辑做复杂了。

更现实的做法通常是：

1. 先让场景依赖链和目录结构保持清晰。
2. 高频复用资源优先吃缓存收益。
3. 只对少数大场景、一次性资源、动态资源做显式清理。

也就是说，手工资源管理应该是针对问题加上的策略，不应该先默认成全项目常态。

## 最容易混淆的五件事

1. 预加载的是场景资源及其依赖链，不是提前创建场景实例。
2. 第二次打开更快，通常是资源缓存复用，不是复用了上次那个场景实例。
3. `close()` 处理的是场景打开状态，不等于资源已经释放。
4. `destroy()` 处理的是实例生命周期终点，不等于缓存已经清空。
5. `clearRes()` 处理的是资源缓存，不负责替你判断谁还在使用这些资源。

## 结论

处理场景资源时，最重要的不是多记几个 API，而是先把三条线分开：

1. 用场景依赖链理解“哪些资源会跟着场景一起加载”。
2. 用缓存复用理解“为什么预加载和第二次打开会更快”。
3. 用 `close()`、`destroy()`、`clearRes()` 区分实例退出、实例结束和资源清理。

把这三条线分开后，再决定哪些资源交给默认缓存复用，哪些场景需要手工回收，判断会稳很多。
