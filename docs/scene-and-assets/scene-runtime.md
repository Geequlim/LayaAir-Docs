# 场景文件在运行时是怎么生效的

很多人第一次看到：

```ts
await Laya.Scene.open("scenes/Main.ls");
```

会把它理解成“把一个已经做好的画面直接显示出来”。

这不准确。

场景文件在运行时真正扮演的角色，更接近：

1. 它先是一份层级描述数据
2. 运行时把这份数据加载进来
3. 再把它反序列化成场景实例和节点树
4. 最后把这个场景实例接入当前运行时树

如果这四步不分开看，就很容易把下面几件事混成一件事：

1. 场景文件本身是什么
2. `Laya.Scene.open()` 到底做了什么
3. 场景什么时候真的进入舞台
4. 场景关闭后，资源为什么不一定立刻消失

## 先看最小主链

最小例子可以先写成这样：

```ts
await Laya.init({
  designWidth: 1334,
  designHeight: 750,
  scaleMode: "showall"
});

await Laya.Scene.open("scenes/Main.ls");
```

从运行时语义看，这行 `open()` 更接近下面这条主链：

1. 把 `.ls` 当成场景层级资源去加载
2. 解析场景依赖的贴图、材质、子层级资源等内容
3. 根据场景数据创建对应的运行时场景实例
4. 把场景实例加入运行时树
5. 执行场景打开后的生命周期逻辑

所以 `Laya.Scene.open()` 不是“只做显示”，而是把加载、实例化、接树、打开这几步串起来的高层入口。

## 场景文件本质上表示什么

场景文件不是一张截图，也不是一个已经活着的对象实例。

它更像一份“如何构造这个场景”的说明数据。

这份数据通常至少描述这些内容：

1. 场景根节点和子节点的层级关系
2. 每个节点的类型和属性值
3. 节点上挂了哪些组件或脚本
4. 节点引用了哪些贴图、材质、字体、音频或其他资源
5. 如果声明了运行时类，实例化时应该恢复成什么运行时对象

因此，`.ls` 文件保存的不是“运行中的场景对象”，而是“让运行时重新创建这个场景对象所需的数据”。

这也是为什么：

1. 场景文件能被重复打开多次
2. 每次打开拿到的是新的场景实例
3. 场景数据相同，不等于运行时状态也相同

例如：

```ts
const sceneA = await Laya.Scene.open("scenes/Main.ls");
const sceneB = await Laya.Scene.open("scenes/Main.ls");
```

这更接近“根据同一份描述数据创建了两次场景”，而不是“取回了同一个已经存在的场景对象”。

## `Laya.Scene.open()` 在运行时做了什么

只看使用层，`Laya.Scene.open(url)` 可以理解成一个面向场景的高层流程封装。

它背后至少包含两段逻辑。

### 第一段：把场景资源变成可实例化结果

这一步和普通 `Laya.loader.load()` 的层级资源链路是同一类问题。

也就是说，运行时会先处理：

1. 识别这是一个场景资源
2. 加载场景数据本身
3. 继续加载它依赖的资源
4. 准备好生成运行时场景实例所需的信息

这一段的重点不是“画面出现了”，而是“场景已经可以被创建”。

### 第二段：把场景实例真正打开

场景资源准备好后，运行时才会继续做实例级的打开过程。

从语义上看，这一步主要包括：

1. 创建场景实例和它内部的节点树
2. 按打开参数决定是否关闭其他场景
3. 把当前场景接入场景根节点
4. 如果场景带有 3D 部分，再把对应内容接入舞台中的 3D 运行结构
5. 触发场景打开后的生命周期回调

因此，`open()` 的返回值之所以有意义，不是因为它只是“加载完成了”，而是因为场景实例已经走到了可参与运行时更新和显示的阶段。

## 场景实例是怎么加入运行时树的

理解场景运行时行为，最关键的一条边界是：

场景文件不会自己出现在舞台上，真正进入运行时树的是“由场景文件创建出来的场景实例”。

可以把关系先记成两层：

1. `Laya.stage` 是全局舞台入口
2. 场景实例是后续被挂到舞台体系里的显示层级对象

所以这句代码：

```ts
await Laya.Scene.open("scenes/Main.ls");
```

从运行时意义上，不是“让 `.ls` 文件挂到舞台”，而是“让 `.ls` 描述出的那个场景实例加入当前运行时树”。

这也是为什么打开场景后：

1. 场景内节点开始参与显示列表
2. 场景内组件和脚本开始进入运行节奏
3. 场景里的输入、动画、计时和显示行为才真正有了宿主

如果只是拿到了场景资源，但没有完成实例化和接树，这些运行时行为都还不会成立。

## `open()` 和 `load()` 的边界要分开看

最容易混淆的一点是把 `Laya.Scene.open()` 和 `Laya.loader.load()` 当成一回事。

它们有关，但不相同。

```ts
const res = await Laya.loader.load("scenes/Main.ls");
```

这更偏“把场景资源链路加载完成”。

而：

```ts
await Laya.Scene.open("scenes/Main.ls");
```

这更偏“把场景作为运行中的内容真正打开”。

可以先用一句话区分：

1. `load()` 解决资源进入运行时的问题
2. `open()` 解决场景进入运行时树的问题

对场景主题来说，第二步才是“生效”的关键。

## 为什么场景打开后脚本和组件才开始工作

场景实例在真正加入运行时树之前，层级数据只是“已经能创建对象”。

只有接入运行时树之后，下面这些事才有了完整上下文：

1. 显示列表更新
2. 舞台尺寸与坐标体系参与计算
3. 输入事件开始沿当前对象树分发
4. 依赖场景宿主的脚本生命周期开始推进

因此，很多“资源明明加载了，为什么脚本没反应”的问题，根因不是资源没到，而是场景实例还没真正进入运行阶段。

## 场景生命周期和资源生命周期不是一回事

场景实例的打开/关闭和资源的加载/缓存是两条独立的链：`scene.close()` 结束的是场景实例的打开状态，它依赖过的贴图、材质等资源是否仍留在缓存里，要由资源系统单独决定。因此会出现“场景关了但内存没降”“第二次打开更快”等现象。

关于 `close()`、`destroy()` 和 `clearRes()` 各自的职责边界、资源何时真正进入清理阶段，以及场景与资源互相影响的具体机制，见 [scene-assets.md](./scene-assets.md)。

## 一个更接近真实使用的最小例子

```ts
async function bootstrap() {
  await Laya.init({
    designWidth: 1334,
    designHeight: 750,
    scaleMode: "showall"
  });

  const scene = await Laya.Scene.open("scenes/Main.ls");

  Laya.timer.once(3000, null, () => {
    scene.close();
  });
}

bootstrap();
```

这段代码里每一步的角色分别是：

1. `Laya.init(...)` 建立引擎运行时
2. `Laya.Scene.open(...)` 把场景资源变成已打开的场景实例
3. 场景实例加入运行时树后，内部节点和脚本才真正开始参与运行
4. `scene.close()` 结束的是这个场景实例的打开状态，不是直接等于所有相关资源都被立即销毁

## 最容易混淆的五件事

1. 场景文件保存的是层级描述数据，不是已经存在的场景实例
2. `Laya.Scene.open()` 不只是加载文件，还包含实例化和打开场景
3. 真正加入运行时树的是场景实例，不是 `.ls` 文件本身
4. 场景生命周期和资源生命周期有关联，但不是同一条生命周期
5. 场景关闭后，资源是否仍在缓存中，要看资源系统后续状态

## 结论

理解“场景文件在运行时怎么生效”，最重要的不是把场景当成一个静态文件看，而是先建立这条稳定主链：

1. 场景文件先表示一份层级描述数据
2. `Laya.Scene.open()` 先加载这份数据及其依赖资源
3. 运行时再把它反序列化成新的场景实例
4. 场景实例加入运行时树后，显示、脚本和交互才真正开始生效
5. 场景关闭处理的是实例状态，资源是否释放还要看资源生命周期本身

把这五步分开之后，再看启动场景、切场景、资源复用和场景关闭问题，会清楚很多。
