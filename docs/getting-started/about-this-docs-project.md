# 这份文档项目和官方文档是什么关系

这份文档项目不是官方文档的镜像，也不是对官方文档的逐页改写。

当前第一阶段内容默认以 `LayaAir 3.4` 为准，不主动展开跨版本差异。

更准确地说，它和官方文档是同一套引擎与项目模型下的两种分工：

1. 官方文档负责更完整的能力覆盖、功能说明、API 与常规用法。
2. 这份文档项目负责把 `LayaAir 3.4` 的标准项目语义重新讲清楚，重点放在代码、资源、运行时和项目结构之间的关系上。

所以两者不是互相替代，而是互相补充。

## 这份文档项目主要解决什么问题

很多初学者第一次接触 LayaAir 时，最容易混淆的不是“某个按钮在哪里”，而是下面这些问题：

1. 标准 Laya 项目到底由哪些目录和文件构成。
2. `src/`、`assets/`、`.laya`、场景、预制体、发布产物之间是什么关系。
3. 引擎启动、项目入口、场景打开、资源加载各自属于哪一层。
4. 预览能跑、发布后出错时，问题更可能出在哪一层。

这份文档更关注这些长期稳定的问题，而不是把内容主要组织成一组编辑器界面操作说明。

## 为什么这里会采用 code-first 写法

这里说的 code-first，不是“只写代码，不用场景、不用资源、不用标准项目”。

它真正强调的是：先按代码和运行时语义理解项目，再去看场景、资源和工具链行为。

原因很直接：

1. 代码入口、资源路径、场景引用关系、发布结果，决定了项目真正如何运行。
2. 这些东西比界面按钮位置更稳定，也更接近排错时真正要看的对象。
3. 团队协作、版本管理、自动化构建，最终也都是围绕文件、目录、资源和脚本发生的。

所以这份文档会优先解释：

1. 项目里有哪些关键文件。
2. 运行时到底执行了什么。
3. 某个资源为什么能被加载，或为什么发布后失效。
4. 一个标准项目从启动到进入首个场景，链路是怎样接起来的。

## 为什么这里会强调 agent-friendly

这份文档不只是给人读，也希望足够适合 agent、LLM 和自动化工具理解。

要做到这一点，文档结构就不能过度依赖“先点哪里、再看哪个面板、再切到哪个标签页”这类界面顺序。

因为对 agent 来说，真正稳定、可推理、可引用的是：

1. 文件路径和目录职责。
2. 类、函数、资源、场景之间的关系。
3. 启动顺序、加载顺序、发布约束。
4. 概念边界，例如“引擎启动”和“项目入口”不是一回事。

这也是为什么这里的文章会尽量围绕项目模型来组织，而不是围绕编辑器 UI 来组织。

## 为什么不按编辑器 UI 来组织整套内容

不是因为编辑器不重要，而是因为“界面位置”通常不是最稳定的知识层。

同一个问题，如果只记成某个按钮在哪，读者很容易在下面几种情况下重新迷路：

1. 编辑器版本变化后，按钮位置变了。
2. 团队成员并不总是通过同一种界面进入同一个功能。
3. 真正排查问题时，最后还是要回到代码、资源、路径和运行行为本身。

所以这里更倾向于把内容写成：

1. 这个功能在项目里对应什么文件或资源。
2. 它在运行时会影响什么。
3. 它在预览、构建、发布时可能带来什么差异。

这样的组织方式，对人类读者和 agent 都更稳定。

## 这并不意味着脱离标准 Laya 项目流程

这份文档虽然不是围绕编辑器界面来写，但讨论的仍然是标准 `LayaAir 3.4` 项目流程。

也就是说，本文档默认仍然承认这些基础约定：

1. 项目是可被 LayaAir 识别的标准工程。
2. 业务代码主要放在 `src/`。
3. 开发期资源主要放在 `assets/`。
4. 场景、预制体、资源引用、预览、构建、发布，仍然遵守标准项目链路。

区别只在于：这里会优先解释这些流程背后的项目语义，而不是把重点放在界面操作步骤上。

## 应该把两类文档怎么配合使用

可以把分工简单理解成这样：

1. 需要查完整功能覆盖、组件说明、API 或特定工具能力时，优先看官方文档。
2. 需要搞清楚标准项目结构、启动链、资源组织、运行时边界和发布语义时，优先看这份文档项目。

如果一个问题同时涉及两边，也不矛盾。

例如，官方文档适合回答“这个能力怎么用”，而这份文档更适合回答“它在标准项目里应该放在哪一层理解”。

## 结论

这份文档项目可以把自己定位成：基于 `LayaAir 3.4` 标准项目模型的一套代码优先、结构优先、运行时优先的解释型文档。

它不替代官方文档，但会尽量把那些最容易在真实项目中混淆的边界讲清楚，而且尽量写成人和 agent 都能稳定复用的形式。
