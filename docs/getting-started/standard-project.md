# 什么是标准 Laya 项目

本文只讨论 `LayaAir 3.4`。

这里说的“标准 Laya 项目”，不是指必须按某个样例工程逐字照抄，而是指遵守 `LayaAir 3.4` 默认项目约定的一类项目结构：

1. 根目录有 `.laya` 项目识别文件。
2. 业务代码主要放在 `src/`。
3. 开发期资源主要放在 `assets/`。
4. 运行时加载路径、场景资源、发布产物都围绕这套结构组织。

它仍然适用于不依赖 IDE 界面的开发流程。你可以直接编辑代码、组织资源、运行命令行构建，但项目本身依然是 LayaAir 能识别和处理的标准工程。

## 最少要有哪些部分

一个可工作的标准项目，至少要有下面几类内容：

```text
my-project/
  MyProject.laya
  src/
    GameMain.ts
  assets/
    resources/
      images/
        logo.png
  bin/
```

这几个部分里，真正决定项目是否能继续开发的核心只有三类：

1. `.laya`：说明这是一个 Laya 项目。
2. `src/`：放你的脚本和业务逻辑。
3. `assets/`：放开发期要管理的资源。

`bin/` 也常会出现，但它不是你日常组织业务资源的主目录。

## 根目录里最重要的文件和目录

### `.laya` 是项目识别文件，不是运行入口

`.laya` 文件位于项目根目录，文件名通常就是项目名。

它的职责是让工具链把这个目录识别为一个 LayaAir 项目，并记录项目层面的基本信息。它不是运行时入口，也不是等价于 `main.ts`、`index.ts` 这类 Web 工程入口文件。

这点很重要，因为很多旧经验会把“根目录里的某个 TS 文件”误当成固定入口。但在标准 Laya 项目里，项目能否启动，不是由 `.laya` 文件直接执行决定的。

### `src/` 是代码主目录

`src/` 是日常开发最重要的目录之一。

通常这里会放：

1. 启动脚本。
2. 场景运行时脚本。
3. 通用组件、工具函数、管理类。
4. 游戏或应用业务逻辑。

例如：

```text
src/
  GameMain.ts
  game/
    PlayerController.ts
  ui/
    HudController.ts
```

需要注意的是，`Main.ts` 不是标准项目必须存在的固定文件名。某些模板或样例里会有 `Main.ts`，但那只是默认示例，不应把它理解为规范本身。

### `assets/` 是开发期资源主目录

`assets/` 是另一个最高频目录。场景、预制体、图片、音频、材质、图集、配置资源，通常都从这里组织。

从开发视角看，可以先把它理解为“项目资源总目录”。

例如：

```text
assets/
  scenes/
  prefab/
  materials/
  textures/
  resources/
    ui/
    audio/
```

但这里还要再区分一层：

1. `assets/` 解决的是项目内资源管理问题。
2. `resources/` 更接近一种“强制打包”的便利目录约定。

也就是说，`assets/` 是大范围概念，`assets/resources/` 只是其中一种特殊约定，不应该被误解成“代码加载资源的唯一正确位置”。

### `resources/` 更像强制打包目录

`resources/` 最大的意义，不是“只有放进去代码才能加载”，而是它像 Unity 的 `Resources` 一样，提供了一种更省心、但也更粗放的强制打包方式。

这类做法的特点是：

1. 不要求先建立清晰的场景依赖关系
2. 更适合快速确保某些资源一定会进入产物
3. 但也更容易把资源边界做得越来越模糊

所以它可以用，但一般不建议把它当成默认策略。

最小示例：

```ts
await Laya.loader.load("resources/ui/logo.png");
```

对应的项目内资源位置通常是：

```text
assets/resources/ui/logo.png
```

这样讲更接近引擎和项目工具链的实际语义：

1. 开发时文件在 `assets/` 下管理。
2. 运行时代码可以加载任意最终会进入产物且路径可访问的资源，不是只有 `resources/...` 才能被加载。
3. 使用 `resources/` 的主要目的，是更直接地确保这批资源会被带进产物。

### `bin/` 不是常规资源根目录

`bin/` 经常让初学者误解，因为预览或调试时会直接看到它，里面也可能有 `index.html` 或其他运行产物。

但在标准 Laya 项目里，`bin/` 更接近预览期或中间产物目录，不应作为日常资源组织的主位置。

有两个结论要记住：

1. 不要把 `bin/` 当成项目资源源目录。
2. 不要把普通项目资源长期放到 `bin/` 里维护。

如果把资源直接堆进 `bin/`，短期可能“看起来能跑”，但会很容易和标准资源管理、发布收集规则、团队协作方式脱节。

## 哪些目录是日常高频使用的

高频目录其实很少，通常就是下面几个：

### `src/`

处理代码结构、运行时逻辑、脚本绑定、启动流程。

### `assets/`

处理场景、预制体、贴图、音频、材质、配置资源。

### `assets/resources/`

处理那些你希望直接强制打包、并接受它以较粗粒度方式进入产物的资源。

如果你的开发流程以代码为主，这三个位置已经覆盖了大部分日常工作。

## 哪些目录知道作用即可

除了上面几个高频目录，根目录下可能还会有一些配置文件、构建脚本、包管理文件，或者预览与发布生成的目录。

这一类内容当然重要，但它们通常不需要你每天都手动维护资源内容本身。

对入门阶段来说，更实用的判断方式是：

1. 改业务逻辑，先看 `src/`。
2. 改资源组织，先看 `assets/`。
3. 查预览结果或中间产物，再看 `bin/`。

## 对运行真正有影响的配置点

理解标准项目时，不要把“目录存在”误当成“运行行为已确定”。真正影响运行的，通常是下面几类点：

### 启动代码

引擎是否初始化、舞台如何配置、是否打开某个场景，最终都要落到代码和启动配置上。

例如，`Laya.init()` 属于引擎启动，而不是 `.laya` 文件的职责：

```ts
await Laya.init(0, 0);
```

### 启动场景或启动脚本

标准项目的“项目入口”更接近启动场景以及它绑定的运行时脚本，而不是固定某个 `Main.ts` 文件名。

这也是为什么样例里的入口写法只能当样例，不能直接当项目规范。

### 资源引用关系

一个资源是否会在运行时可用，不只取决于它是不是放在 `assets/` 下，还取决于：

1. 它是否被场景、预制体等层级资源引用。
2. 它是否会被代码通过可收集的路径规则加载。
3. 它是否符合当前发布目标的资源收集规则。

所以“文件在 `assets/` 里”不等于“发布后一定可用”。

## 对发布真正有影响的配置点

发布阶段最容易出错的，不是代码写法本身，而是对资源进入产物的判断。

至少要避免这三个误解：

1. `assets/` 下所有文件都会自动进入发布包。
2. 只要预览时能访问，发布后就一定还能按同样路径访问。
3. `bin/` 里的文件布局就是项目长期稳定的资源布局。

更稳妥的理解方式是：

1. `assets/` 是开发期资源管理根。
2. 运行时真正访问哪些资源，要看场景引用和代码加载路径。
3. 发布产物收集哪些文件，要看发布规则，而不是只看目录名。

## 推荐的最小资源布局

如果你希望项目结构更容易维护，可以从下面这种简单约定开始：

```text
assets/
  scenes/
    MainScene.ls
  prefab/
    Enemy.lh
  resources/
    ui/
      hud.atlas
    audio/
      hit.wav
```

这种布局的好处是：

1. 场景与预制体资源单独放。
2. `resources/` 适合少量“明确要强制打包”的资源，不适合无脑兜底整个项目。
3. 后续排查“为什么这个资源没被加载到”时，边界更清楚。

## 常见误区

### 误区 1：把 `Main.ts` 当成固定标准入口

不是。`Main.ts` 可能存在，但它不是标准项目语义里唯一且固定的入口名。

### 误区 2：把 `bin/` 当成资源根目录

不是。`bin/` 更像预览或中间产物位置，常规资源应以 `assets/` 为源。

### 误区 3：以为 `assets/` 下所有文件都会自动发布

不是。资源是否进入产物，要看引用关系、加载方式和发布规则。

### 误区 4：不依赖 IDE 就等于不能用标准项目

也不是。标准项目描述的是项目结构和资源组织方式，不等于必须依赖某个界面操作流程。

## 结论

标准 Laya 项目可以先抓住一句话：它是一种以 `.laya` 识别项目、以 `src/` 管代码、以 `assets/` 管资源的默认工程结构。

再记住四个边界：

1. `.laya` 是项目识别文件，不是运行入口。
2. `src/` 和 `assets/` 是开发主战场。
3. `resources/` 更像强制打包目录，不是代码加载的唯一入口。
4. `bin/` 不是常规资源根目录。

如果这四点没有混淆，后面的启动流程、资源加载和发布规则就会容易理解很多。
