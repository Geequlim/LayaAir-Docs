# 文本、图片和图集应该怎么管理

UI 资源最容易混乱的地方，不是 API 不会用，而是把几类完全不同的东西混成了一类：

1. 文本是运行时生成的内容，重点在字体选择和更新成本。
2. 图片是独立纹理资源，重点在加载路径、复用关系和尺寸边界。
3. 图集是成组纹理组织方式，重点在“哪些图应该放一起”，而不是“图越多越好”。

如果这三类东西不先分开看，常见结果就是：能显示，但后续难维护、难发布、也难优化。

## 先分清三种资源职责

在 2D UI 里，可以先用下面这条最实用的边界来判断：

1. 会频繁变化的文案，优先保留为 `Laya.Text` 这类运行时文本。
2. 会复用的独立图片，按业务目录管理为普通贴图或图集资源。
3. 会反复一起出现的一组小图，再考虑做成图集。

这和资源目录文档里的思路是一致的：先按使用关系组织，再决定要不要放进 `resources/` 作为动态加载入口。

## 文本先解决“要不要做成图片”

很多 UI 问题，一开始就选错了文本方案。

最常见的三种做法是：

1. 系统字体或注册字体配合 `Laya.Text`
2. 位图字体
3. 直接把字做进图片

这三种方案没有谁绝对更高级，关键是内容变化方式不同。

### 适合继续用普通文本的情况

如果文本内容满足下面这些特点，优先保留为普通文本：

1. 文案会变
2. 需要多语言
3. 需要动态拼接数字、玩家名、倒计时
4. 需要换行、对齐、截断等排版能力

最小例子：

```ts
const scoreText = new Laya.Text();
scoreText.font = "Arial";
scoreText.fontSize = 32;
scoreText.color = "#ffffff";
scoreText.text = "Score: 1280";
Laya.stage.addChild(scoreText);
```

这类文本的重点不是放进图集，而是：

1. 字体是否稳定可用
2. 样式是否统一
3. 更新频率是否过高

如果一个计时器每帧都改整段长文本，问题通常不在字体，而在更新方式本身太重。

### 什么时候考虑位图字体

位图字体更适合样式稳定、字符集合可控、同时又不想把每个字都做成单独图片的情况。

常见场景：

1. 分数
2. 伤害数字
3. 倒计时
4. 只有少量字符的标题字

例如：

```ts
await Laya.loader.load("resources/fonts/score.fnt", Laya.Loader.FONT);

const score = new Laya.Text();
score.font = "score";
score.text = "1280";
Laya.stage.addChild(score);
```

这里的核心收益是：

1. 字形外观稳定
2. 数字类文本更容易统一视觉
3. 字符集有限时，资源边界更可控

但它的边界也很明确：

1. 不适合正文型长文案
2. 不适合字符集很大的多语言文本
3. 增一个新字符，通常就要回到字体资源本身

### 不要轻易把文字烘焙成图片

把文字直接做进图片，只有在下面几种场景里才通常值得：

1. 纯装饰标题
2. 完全不会改的 logo 文案
3. 特效字样本身就是一张美术资源

如果普通按钮文案、列表标题、活动描述也都做成图片，后面通常会同时遇到四个问题：

1. 多语言资源数量暴涨
2. 改字必须改图
3. 搜索、排版、截断能力消失
4. 很多本来能共用一个样式的文本，被拆成大量离散图片

## 字体选择先看稳定性，再看风格

字体管理里最常见的误区，是一上来追求“特别的字”，却没先确认运行时是否稳定。

更稳的顺序通常是：

1. 先选一套项目主字体，用于大部分正文和按钮文案
2. 再给数字或装饰标题补少量位图字体
3. 最后才处理非常特殊的艺术字图片

如果项目需要自定义字体文件，重点不是“能不能加载”，而是“这一套字体是否真的值得全局依赖”。

这类字体通常需要先进入项目资源，再按运行时可用的字体名用于 `Laya.Text`。这里真正该先确认的是：字体是否稳定可用，以及是否真的需要全局引入这一份资源。

对组织层面，更实用的约定是：

1. 正文和按钮文案尽量少换字体家族
2. 数字和标题才单独分出字体方案
3. 不要让同一界面同时混用太多字体资源

字体种类一多，不只是视觉不统一，资源加载和回归检查也会一起变复杂。

## 图片先按“是否独立复用”来分

图片资源是否该单独放，先不要看它大不大，先看它是不是独立复用。

### 适合单独作为普通图片的情况

下面这些资源更适合继续做成独立图片：

1. 大背景
2. 大幅插图
3. 全屏遮罩
4. 经常单独加载或单独释放的图片

例如：

```ts
const bg = await Laya.loader.load("resources/ui/login/bg.png");

const sprite = new Laya.Sprite();
sprite.texture = bg;
Laya.stage.addChild(sprite);
```

这类图片和图集的组织重点不同。它们通常不需要和一堆小图硬塞到同一个包里。

### 适合进入图集的情况

如果一批图片满足下面的特征，就更适合打成图集：

1. 经常一起出现在同一个界面
2. 单张都比较小
3. 生命周期接近
4. 会被多个 `Image`、`Sprite` 反复取用

典型例子：

1. 按钮皮肤
2. 图标
3. 小红点
4. 面板边角和装饰件

图集的真正价值不是省目录，而是让“一批一起使用的小图”作为一组资源被加载和管理。

## 图集按界面和生命周期分组，不按美术工种分组

很多项目的第一批图集，都会犯一个错误：

1. 把所有按钮做成一个图集
2. 把所有图标做成一个图集
3. 把所有 UI 小图做成一个超级图集

这会导致加载边界越来越差。

更稳的分法通常是：

1. 登录界面一组
2. 大厅界面一组
3. 战斗 HUD 一组
4. 公共小组件一组

也就是优先按“谁会一起加载、一起常驻、一起释放”来拆。

例如：

```text
assets/
  resources/
    ui/
      common/
        common.atlas
        common.png
      login/
        login.atlas
        login.png
      battle/
        battle-hud.atlas
        battle-hud.png
```

对应加载时也更清楚：

```ts
await Laya.loader.load("resources/ui/login/login.atlas");

const logo = Laya.loader.getRes("resources/ui/login/logo.png") as Laya.Texture;
```

这里和普通图片加载的区别是：你加载的是图集入口，真正使用时拿的是图集里的小图路径。

## 图集不是越大越好

把图集做得很大，短期看可能觉得“省事”，但通常会带来另外三类问题：

1. 某个小图一改，整张图集都可能变化
2. 只想进一个小面板，却被迫连整大组资源一起加载
3. 一张大图集被多个界面勉强共用，释放边界变差

所以图集大小的判断，重点不只是像素尺寸，更是资源边界是否合理。

对于 UI 来说，更值得优先追求的是：

1. 同屏常用的小图放一起
2. 不同流程的大图不要硬合
3. 变化频率不同的资源尽量拆开

例如活动页资源经常改版，就不该长期和常驻 `common` 图集合在一起。

## 动态图集解决的是运行时碎片，不是资源组织混乱

动态图集最容易被误解成“可以不用好好打图集了”。

这不准确。

动态图集的价值更接近：运行时把符合条件的小纹理再临时收拢，减少零散纹理带来的渲染成本。

如果前期资源组织已经很乱，动态图集不会替你自动修正：

1. 哪些资源应该一起加载
2. 哪些资源应该独立释放
3. 哪些资源本来就不该反复切换

所以它是后续优化手段，不是前期资源管理方案。

## 动态图集什么时候值得开

更适合考虑动态图集的场景通常是：

1. 运行时会产生不少零散小图
2. 小图来源比较分散，静态图集很难提前覆盖完全
3. 同屏元素很多，切换纹理带来的成本已经明显可见

例如：

1. 大量可复用小图标
2. 动态创建的头像框、徽章、角标
3. 由多批资源拼起来的滚动列表单元

这类场景里，动态图集才可能真正带来收益。

## 动态图集的边界

动态图集不是对所有图片都有效。

无论你启用的是自动动态图集能力，还是手动管理一组动态图集，最需要先建立的边界有四条：

1. 它更适合小图，不适合大背景和大插图
2. 它解决的是运行时合并，不会替代静态图集的加载边界设计
3. 频繁变化、频繁销毁的大量内容，不一定会因为开了动态图集就自动更优
4. 是否值得启用，最终要看实际渲染表现，而不是只看“功能存在”

从引擎实现上看，自动动态图集也会受尺寸阈值约束。默认更接近“只收小纹理”，而不是“所有贴图都往里塞”。

## 动态图集最常见的误用

### 误用 1：拿它代替静态图集规划

如果登录页、战斗页、活动页的小图全是散的，再指望运行时自动合并来兜底，本质上只是把管理问题拖到运行时。

### 误用 2：把大图也当动态图集候选

大背景、长条 banner、整页插画，本来就不属于动态图集最擅长处理的范围。

### 误用 3：一看到 drawCall 就先开动态图集

drawCall 变高不一定只和纹理分散有关，也可能是节点结构、文本更新、遮罩、缓存策略等别的问题。

### 误用 4：忽略文本本身的渲染特点

普通文本、位图字体、图片字，其渲染路径并不完全一样。不要把所有“看起来像 UI”的东西都当成同一种纹理问题。

## 最容易出问题的几种组织方式

### 1. 所有 UI 图都放进 `common.atlas`

短期看很好找，长期通常最难拆。

### 2. 把动态文本都改成图片

这会把多语言、排版和改字成本全部放大。

### 3. 每个按钮单独一张图

如果这些按钮总是同屏出现，后面很容易产生一堆零散小纹理。

### 4. 同一界面同时混很多字体方案

这会把视觉规范、资源体积和测试复杂度一起抬高。

### 5. 只按文件类型分目录，不按界面分资源组

例如只有 `icons/`、`buttons/`、`titles/`，却看不出登录、大厅、战斗分别依赖什么资源。

## 一个更稳的 2D UI 约定

如果团队还没有明确约定，可以先从这个粒度开始：

```text
assets/
  resources/
    ui/
      common/
        common.atlas
        common.png
      login/
        login.atlas
        login.png
        bg.png
      battle/
        hud.atlas
        hud.png
    fonts/
      score.fnt
      score.png
      ui.ttf
```

它背后的判断标准是：

1. 会一起出现的小图放进同组图集
2. 大图单独放
3. 动态文本优先保留为文本
4. 数字类或装饰性强的文本，再单独考虑位图字体

## 结论

管理 2D UI 资源时，最实用的顺序不是“先想怎么优化”，而是：

1. 先决定这是不是文本
2. 再决定它是独立图片还是一组图集小图
3. 最后才决定是否需要动态图集参与运行时优化

如果文本、普通贴图、静态图集、动态图集这四层职责清楚，资源目录、加载方式和后续优化通常都会稳定很多。
