# 一个复杂 UI 应该怎么拆

复杂 UI 最怕的不是节点多，而是把三类东西混在一起：

1. 固定结构。
2. 可复用局部。
3. 当前这一次实例才有的状态。

一旦这三类边界不清楚，后面通常会出现这些问题：

1. 一个页面的节点层级越来越深。
2. 明明只是换一块内容，却要改整页结构。
3. 列表项、弹窗、页签内容反复复制。
4. 运行时状态直接写回结构，导致复用困难。

更稳妥的拆法是先按职责分层，再按复用边界拆成局部单元，最后把变化交给数据和实例状态。

## 先按层拆，不要一上来按控件拆

复杂 UI 更适合先拆成几层职责：

1. 页面外壳层：负责整页的总容器、背景、主区域分块。
2. 功能区块层：负责头部、侧栏、内容区、底栏、弹窗层。
3. 可复用模块层：负责头像卡、奖励条目、任务条目、通用按钮组。
4. 数据项层：负责列表里的单个 cell。

一个常见结构可以先长成这样：

```text
PageRoot
  Bg
  Header
  Content
    LeftNav
    MainArea
  PopupLayer
  ToastLayer
```

这里的重点不是节点名，而是先让每一层只回答一个问题：

1. 这块是不是页面骨架。
2. 这块是不是可单独替换的功能区。
3. 这块是不是会重复出现的局部单元。
4. 这块是不是只为了展示一条数据。

如果一开始就按“这里放几个 `Image`、那里放几个 `Label`”去堆，最后往往会得到一个能显示但很难维护的界面树。

## 页面骨架适合保留稳定层级

页面最外层通常适合用 `Laya.Box` 一类容器把结构先固定下来。

例如：

```ts
const page = new Laya.Box();

const header = new Laya.Box();
const content = new Laya.Box();
const popupLayer = new Laya.Box();

page.addChild(header);
page.addChild(content);
page.addChild(popupLayer);
```

这类骨架层更适合承载：

1. 固定区域划分。
2. 稳定的显示顺序。
3. 弹窗层、提示层这类全页级覆盖层。
4. 大多数实例都不会变化的默认布局。

不适合把页面骨架层直接当成业务状态仓库。比如当前选中页签、背包数据、倒计时剩余秒数，都不应该塞进“结构定义”本身。

## 哪些块应该单独拆出来

判断一个局部是否值得单独拆，先看这三件事：

1. 它会不会重复出现。
2. 它会不会在多个页面复用。
3. 它有没有稳定的内部层级和默认资源配置。

只要有两条明显成立，就很适合独立成局部模块，很多时候也适合继续做成 prefab。

常见适合拆开的块：

1. 顶部资源栏。
2. 页签按钮组。
3. 角色信息卡。
4. 奖励条目。
5. 商品条目。
6. 通用确认弹窗。

常见不必急着拆的块：

1. 只在当前页面出现一次的简单装饰节点。
2. 只有两三个子节点的一次性说明区。
3. 结构本身主要由运行时数据临时拼出来的区域。

## prefab 的边界要落在“稳定局部”，不要落在整页一把包

prefab 更适合承接稳定局部，不适合把整个复杂页面无差别打成一个大资源，再在运行时到处局部覆盖。

更常见的拆法是：

1. 页面本身是一层主结构。
2. 其中反复出现或职责独立的部分做成 prefab。
3. 列表里的单个单元再单独做成 cell prefab。

例如一个活动页，可以拆成：

1. `ActivityPage`：页面骨架。
2. `RewardBlock`：奖励展示块。
3. `TaskCell`：任务列表单元。
4. `ConfirmDialog`：通用确认弹窗。

短例子：

```ts
const prefab = await Laya.loader.load("resources/prefab/ui/TaskCell.lh") as Laya.Prefab;
const cell = prefab.create() as Laya.Box;
```

这里更重要的不是“少写了多少 `new`”，而是 `TaskCell` 的内部层级、默认图标位、文案位、按钮位都已经稳定下来了。

如果一个 prefab 需要在每个页面里大改子节点增删、重排层级、换大半资源，它通常说明边界拆大了，应该再往下切。

## 列表和滚动容器，不是同一种职责

复杂 UI 里很容易把“能滚动”都混成一类，但 `List` 和滚动容器解决的问题并不一样。

### `Laya.List` 解决重复数据项

当一块内容满足下面条件时，优先考虑 `Laya.List`：

1. 一批同构条目反复出现。
2. 每条数据只是内容不同，结构基本一致。
3. 你希望条目渲染跟着数据走。

例如任务列表、背包格子、邮件列表、排行榜。

最小例子：

```ts
const list = new Laya.List();
list.array = [
  { title: "任务1", progress: 3 },
  { title: "任务2", progress: 7 }
];
```

这类区域的关键不是“有滚动条”，而是“有一批结构一致的数据项”。

### 滚动容器解决视口裁切和内容移动

当一块内容的重点是“一个较大的内容区，需要在有限视口里滚动查看”，更适合滚动容器。

例如：

1. 一整段说明面板。
2. 一个较长的富文本区。
3. 一块自由排布、并不严格同构的内容区域。

这种场景里，关注点通常是：

1. 视口大小。
2. 内容区域大小。
3. 滚动位置。

也就是说：

1. `List` 更像“数据项重复器”。
2. 滚动容器更像“可滚动视窗”。

如果一个区域本质上是重复条目，就不要手工在滚动容器里堆很多相似节点；如果一个区域本质上不是同构条目，也不要勉强套成列表。

## 数据驱动 UI，先让数据决定内容，不要让节点自己记业务

复杂 UI 最稳定的方式，是让结构负责“有哪些槽位”，让数据决定“槽位里显示什么”。

例如奖励块，不要把“第 3 个奖励已经领过”这种业务结论写死在结构里，而是让数据驱动显示：

```ts
const rewardData = { icon: "gold", count: 100, claimed: true };
```

对应的 UI 实例只根据数据刷新：

1. 图标显示什么。
2. 数量显示什么。
3. 是否显示已领取状态。

列表尤其应该这样处理。页面应该保存一组数据，而不是长期手工维护一堆条目节点的状态同步。

例如：

```ts
list.array = questDataList;
```

这样更新任务列表时，先改数据，再让列表刷新，比直接遍历节点挨个改文案更稳定。

## 结构里应该放什么，实例状态里应该放什么

这是复杂 UI 最核心的一条边界。

### 适合留在结构里的内容

1. 稳定的父子层级。
2. 默认尺寸、锚点、位置关系。
3. 默认皮肤、底图、装饰图。
4. 文字槽位、图标槽位、按钮槽位这些固定占位。
5. 大多数实例都一致的默认显隐和初始样式。

### 适合留在实例状态里的内容

1. 当前页面绑定的数据。
2. 当前选中项。
3. 当前页签。
4. 当前滚动位置。
5. 当前倒计时、数量、进度。
6. 当前这一次是否展开、是否已领取、是否可点击。

可以用一句话判断：

如果这个值换一个实例大概率就会不同，它更应该是实例状态，而不是结构默认值。

例如同一个 `TaskCell`：

1. 标题文本框、进度条、领取按钮这些属于结构。
2. “今天登录 1 次”“3/5”“按钮是否可领”这些属于实例状态。

## 页面级状态不要散在子节点里

复杂 UI 还容易出现另一个问题：每个按钮、每个条目、每个局部块都偷偷保存一点业务状态，最后没人知道页面当前到底由谁说了算。

更稳妥的方式通常是：

1. 页面持有页面级数据。
2. 区块持有自己必要的局部显示状态。
3. 单个 cell 只接收当前这一条数据，不长期拥有整页业务真相。

例如：

1. 当前选中的左侧页签，应该由页面或主区块管理。
2. 某个 `TaskCell` 是否高亮，只是它根据当前数据算出来的显示结果。

这样做的好处是页面切换、重刷列表、关闭弹窗时，状态来源比较单一，不容易互相打架。

## 资源布局也要跟拆分边界一致

UI 拆得再清楚，如果资源目录没有跟着边界走，后面仍然会乱。

比较稳的做法通常是：

1. 页面级 prefab 放在 UI prefab 目录下。
2. 通用 cell、弹窗、按钮组放在可复用子目录下。
3. 贴图、图集按 UI 模块或功能区归类。
4. 少量需要代码直接加载的 UI 动态资源，再放到 `resources/` 语义下管理。

例如：

```text
assets/
  prefab/
    ui/
      page/
      common/
      cell/
  textures/
    ui/
      activity/
      common/
  resources/
    ui/
```

重点不是目录名必须完全一致，而是页面骨架、可复用模块、动态加载资源这三条线不要混在一起。

## 一个实用拆法

假设要做一个“活动页面”，可以先这样拆：

1. `ActivityPage`：负责背景、顶部栏、左侧页签区、右侧内容区、弹窗层。
2. `TabButton`：负责单个页签按钮表现。
3. `TaskList`：负责任务列表区域。
4. `TaskCell`：负责单条任务展示。
5. `RewardItem`：负责奖励条目。
6. `ConfirmDialog`：负责确认弹窗。

然后把数据分开：

1. 页面状态：当前页签、当前活动 id。
2. 列表数据：任务数组。
3. 单元数据：标题、进度、奖励、领取状态。

这样后面即使把“任务列表”换成“商店列表”，通常也只是替换右侧区块，而不是重做整页。

## 结论

拆复杂 UI 时，最实用的顺序通常是：

1. 先拆页面层和功能区层。
2. 再把稳定且可复用的局部拆成 prefab。
3. 重复数据交给 `Laya.List` 一类列表结构。
4. 大块可滚动内容交给滚动容器。
5. 让结构保存稳定骨架，让实例状态承接当前数据和交互结果。

只要把“页面骨架”“可复用模块”“数据项”“实例状态”这四层分开，复杂 UI 通常就不会越做越乱。
