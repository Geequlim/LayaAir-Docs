# 什么时候需要关心 `ui2`

很多人第一次看到 `ui2`，会先把注意力放在 `Laya.GBox`、`Laya.GButton`、`Laya.GLabel` 这些新名字上。

但真正需要先分清的，不是哪几个类多出来了，而是：`ui2` 不是在原有 2D UI 上“多一批控件”，而是另一套 UI 组织边界。

如果这个边界没先看清，最常见的结果就是：

1. 明明只需要普通 `Laya.Box`、`Laya.Image`、`Laya.Label` 就够了，却过早切到另一套系统。
2. 一半界面按传统方式搭，一半界面按 `ui2` 思路搭，后面维护越来越别扭。
3. 把 `ui2` 当成“更高级按钮库”，结果没有真正用到它的长处。

## 先把 `ui2` 看成一条系统边界

传统 2D UI 更像是在显示对象树上直接组织界面。

你通常直接处理这些东西：

1. `Laya.Sprite`
2. `Laya.Box`
3. `Laya.Image`
4. `Laya.Label` 或 `Laya.Text`
5. `Laya.List`

重点是：

1. 这棵树怎么搭。
2. 每个节点显示什么。
3. 父子层级、位置、资源和数据怎么配合。

而 `ui2` 更像是在这棵树之上，再引入一套偏组件化、布局化、行为和显示分离的 UI 系统。

它关心的重点会变成：

1. 某个组件是容器、面板，还是行为组件。
2. 布局规则是否应该由组件系统负责。
3. 显示部分和交互部分是否应该拆开。
4. 一组可复用 UI 单元是否应该作为完整组件来组织。

所以判断要不要关心 `ui2`，本质上不是问“我需不需要新控件”，而是问：

我现在是不是已经需要另一套 UI 组织方式。

## 什么时候普通 Laya UI 就够了

下面这些场景，通常继续用普通 2D UI 就够了：

1. 界面规模不大。
2. 结构主要是固定节点树。
3. 你更关心图片、文本、层级和坐标。
4. 交互逻辑并不复杂。
5. 大多数界面都能用 `Laya.Box` 加基础节点清楚表达。

例如一个简单信息面板：

```ts
const panel = new Laya.Box();

const bg = new Laya.Image("resources/ui/panel.png");
const title = new Laya.Label("背包");

panel.addChild(bg);
panel.addChild(title);
Laya.stage.addChild(panel);
```

这类界面的核心问题通常还是：

1. 节点怎么分层。
2. 资源怎么管理。
3. 数据怎么刷新到节点上。

这时先把普通 UI 组织清楚，收益通常比提前切到 `ui2` 更直接。

## `ui2` 真正开始相关的信号

当你的问题开始从“怎么摆节点”变成“怎么组织组件系统”时，`ui2` 才真正开始变得相关。

比较典型的信号有这些。

### 1. 你开始需要更稳定的布局容器语义

如果一个界面里很多区域都不是单纯手摆坐标，而是明确依赖布局规则，`ui2` 会更值得看。

例如你关心的是：

1. 一组子节点按行、列或分页布局。
2. 容器尺寸变化后，子节点要按规则重新排布。
3. 这套排布规则要反复复用。

这时像 `Laya.GBox` 这样的容器就不再只是“另一个 Box”，而是在表达“布局由系统负责”。

### 2. 你开始需要行为和显示分离

普通 UI 里，一个按钮常常会被理解成：

1. 一张底图。
2. 一段文字。
3. 一块点击区域。

而在 `ui2` 里，像 `Laya.GButton` 这种组件更强调行为本身，显示部分可以由别的显示节点来承担。

这类思路适合的不是“随便做个按钮”，而是：

1. 同一类行为组件会反复换皮肤。
2. 同一类交互逻辑要复用到很多不同外观上。
3. 你希望按钮、标签、进度条这些东西更像可组合组件，而不是临时拼装节点。

### 3. 你开始把 UI 当成组件资产，而不只是节点集合

如果你的界面已经出现下面这些情况，`ui2` 的价值会更明显：

1. 同一类模块反复出现。
2. 单个模块内部结构稳定，但样式和内容经常替换。
3. 你希望一个组件既有自己的布局规则，也有自己的交互职责。
4. 你需要更明确地区分容器组件、显示组件、行为组件。

这时问题已经不只是“怎么 addChild”，而是“这个局部到底应该被当成什么组件”。

### 4. 你开始需要一套独立的 UI 根和运行规则

普通 2D UI 多数时候直接挂在场景显示树里理解就够了。

但当你开始依赖 `ui2` 自己的组件体系、面板体系、弹出规则或根节点语义时，你面对的就不再只是普通显示树，而是另一套 UI 运行边界。

这时继续只用“`Sprite` 树怎么搭”去理解问题，通常就不够了。

## 不要把 `ui2` 理解成“传统 UI 的升级版” 

一个常见误区是：

既然 `ui2` 更新，那是不是以后都应该直接用它。

更稳妥的理解通常是：

1. 普通 Laya UI 适合直接、清楚地组织显示对象。
2. `ui2` 适合用另一套组件化边界来组织界面。
3. 它们解决的问题有重叠，但心智模型不完全一样。

所以 `ui2` 不一定是“更高版本的同一件事”，更像是“另一条更组件化的路”。

如果你当前的主要问题还停留在：

1. 不清楚 `Box`、`Image`、`Label` 该怎么配合。
2. 不清楚列表、滚动、资源、层级怎么拆。
3. 不清楚数据和节点状态该怎么分开。

那先把这些基础组织清楚，通常比马上切 `ui2` 更重要。

## 怎么避免两套系统过早混用

最容易出问题的阶段，通常不是完全不用 `ui2`，而是刚知道它存在，就开始局部混搭。

例如：

1. 页面主体还是 `Laya.Box` 树。
2. 中间临时塞几个 `Laya.GButton`。
3. 某个弹窗又改成 `Laya.GBox`。
4. 最后每块界面的组织方式都不一样。

这样做的问题不是“绝对不能运行”，而是边界会越来越不清楚。

更稳的做法通常是：

1. 一个阶段先以普通 UI 为主，把显示树、资源和数据驱动理顺。
2. 只有当某类界面明显需要组件化布局和行为分离时，再单独评估是否切到 `ui2`。
3. 一旦决定某块界面用 `ui2`，尽量让这块界面内部按 `ui2` 的心智模型完整组织，而不是只拿几个类名来混搭。

可以把它理解成：

不是“这个按钮能不能换成 `Laya.GButton`”，而是“这整块 UI 是否值得交给另一套组件系统管理”。

## 一个实用判断顺序

如果你现在在犹豫要不要学 `ui2`，可以先按这个顺序判断：

1. 如果你只是做常规 2D 面板、弹窗、列表，先把普通 Laya UI 用顺。
2. 如果你已经明显遇到布局规则、组件复用、行为显示分离这些问题，再开始看 `ui2`。
3. 如果你准备采用 `ui2`，把它当成独立系统来学，不要只把它当作几个新控件名。

最短结论就是：

普通界面开发时，不必因为 `ui2` 存在就立刻关心它；只有当你的 UI 问题已经跨到“组件系统边界”这一层，它才真正变得重要。
