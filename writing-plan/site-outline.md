# LayaAir 文档站结构大纲

## 目的

这份大纲面向真正的写作落地。

重点不是“把知识点平均分配到若干篇”，而是保证每篇文档都围绕一个清晰的问题展开，不用靠多个关联较弱的名词拼成一篇。

## 组织原则

后续定篇名时，优先采用以下三种组织方式：

1. 一个核心问题
2. 一类核心对象
3. 一条完整工作链

尽量避免这种命名方式：

1. `A、B 与 C`
2. `X、Y 和 Z 概览`
3. 为了凑篇幅把上位主题和边缘主题硬放到一起

例如：

1. “如何启动一个 Laya 项目”是好标题
2. “Laya.init、Config 与 LayaEnv”更适合作为小节，而不是篇名
3. “如何管理场景和预制体”是好标题
4. “场景、预制体与资源依赖”只有在全文主线真的是这条工作链时才适合放在一起

## 参考资料约束

这份大纲默认后续正式文档不会暴露内部参考仓库的信息。

写作时应遵守：

1. 不出现 `references/...` 这类本地路径
2. 不把内部参考文件路径写进正文
3. 如需给外部资料入口，优先使用官方公开地址或开源仓库地址
4. 引用来源保持克制，避免把文档写成资料索引

## 第一版目标

建议第一版控制在 22 到 28 篇。

其中：

1. 主线必写：18 到 22 篇
2. 扩展专题：4 到 6 篇

## 已确认的执行口径

这部分是当前已经确定、后续默认不再反复讨论的写作边界。

1. 版本锁定为 `LayaAir 3.4`，第一阶段不处理跨版本兼容和差异说明
2. 正式文档尽量少外链，只在确有必要时给少量官方公开入口
3. 不写 IDE 操作教程，不以面板、按钮、向导、拖拽流程组织正文
4. 示例代码可以有，但应以最小可理解示例为主，不抄大段代码，不为了“完整”牺牲可读性
5. 术语优先对齐源码语义，再兼顾官方常用说法
